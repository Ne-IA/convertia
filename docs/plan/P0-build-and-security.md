# P0 — Build & Security System

> **The foundation before the foundation.** P0 establishes the guardrail system
> every later phase (P1–P11) runs under: the security concept, the full gate
> system, the build-loop, the dual review, and the test methodology. It is built
> **before** P1 writes any app code, so P1's very first commit already runs through
> the loop + the gates.
>
> Derives from [security-concept.md](../security/security-concept.md) +
> [build-gates.md](../security/build-gates.md). Index: [plan/README.md](README.md).
>
> **This file currently holds the cluster structure + box headings only** — the
> atomic `[ ]` steps are added in the fill pass, after this structure is reviewed
> and signed off. No `[ ]` boxes are filled yet.

## Boundaries (decided, see plan/README.md)

- **P0 ↔ P1:** P0 builds the gate *system* + content-independent gates + the
  framework/wiring-points for language gates; **P1** wires the language-specific
  gates (clippy/tsc/cargo-deny/…) into the P0 framework as it scaffolds, and
  authors the general governance docs.
- **P0 ↔ P10:** P0 *defines* the release-/supply-chain-gate policy; **P10** builds
  the release pipeline. No double-build.
- **Content-independence rule:** a P0 box may only build what can exist without app
  code. Gates that need Rust/TS to act on are *defined + wired-pointed* here and
  *activated* by the producing phase (annotate `→ activated in P1`).

## P0.2 — Gate orchestration framework

**Goal:** the two enforcement planes exist as empty-but-wired harnesses that later
gates plug into.

**Box areas:**
- Git-hook manager (`lefthook`) set up (pre-commit / pre-push / commit-msg), `parallel`, perf budgets; **plane↔hook mapping (chronological): pre-commit = L1, commit-msg = L3 (fires at `git commit`, BEFORE pre-push), pre-push = L2 (fires at `git push`)** — stated so a fill-pass author does not wire commit-msg into pre-push or omit it; **`lefthook` itself is a PINNED + checksum-verified gate tool (§0): the program that executes the whole L1–L3 plane is installed from the §0 pinned-and-checksum-verified set (vendored binary / digest, source-allow-list + signature/≥2-origin corroboration), with a G24 wrong-checksum negative self-test — a poisoned `lefthook` can silently no-op every local hook (the `core.hooksPath`-bypass equivalent) while reporting green**; the **hooks-installed assertion** (G54 — `lefthook install` mandatory after clone, no local protection without it; G54 resolves the EFFECTIVE hooks dir via `git rev-parse --git-path hooks` and asserts `core.hooksPath` is unset/lefthook-managed, so a `core.hooksPath` redirect cannot silently disable all hooks).
- GitHub Actions CI skeleton (L4) — clean-checkout job matrix placeholders (Win/macOS/Linux).
- Release workflow skeleton (L5) — tag-triggered, empty gate slots.
- **CI supply-chain hardening** (content-independent — lives in `.github/`): every workflow declares top-level `permissions: contents: read`, elevated per-job only where needed (the release job touching `MINISIGN_SECRET_KEY`/`MINISIGN_PASSWORD` gets `contents: write` ONLY); every third-party action pinned by **full 40-char commit SHA** (not a tag), kept current via a `dependabot.yml` github-actions entry (its presence asserted by G56); `actionlint` (G49, L1) + `zizmor` (G50, L4) workflow lint; the secret-bearing release job **never runs on a fork pull-request**; per-PUSH-workflow `concurrency: {group, cancel-in-progress: true}` (never on the release/tag workflow) + explicit `timeout-minutes` per job. **`id-token` scope sub-rule (G56):** the workflow-level `permissions:` block sets `id-token` absent/`none` and `id-token: write` appears ONLY on the release/attestation job (an OIDC token is as valuable as a long-lived secret; zizmor's excessive-permissions check targets `GITHUB_TOKEN`, not id-token specifically). **`pull_request_target` safe-handler sub-rule (G56):** no `pull_request_target` workflow without an Environment-protection gate (required reviewers), OR a `pull_request_target` handler that never checks out / runs fork `head_ref` code (the canonical secret-exfiltration vector — G50 flags the trigger but does not constrain the handler shape). **Cache-hygiene lint (G56 sub-rule — build-cache poisoning defence, generalising G37's "an Actions cache is not integrity-protected"):** the secret-bearing release/signing job restores no writable shared cache (cache disabled OR read-only / lockfile+toolchain-derived key); fork-PR workflows share no cache-key namespace with release/`main` (no `restore-keys` prefix bleed); the signed artifact builds `--locked`/`--frozen-lockfile` from a cache-cold or integrity-verified tree — the Cargo/pnpm/sccache/gate-tool/fuzz-corpus caches feed the minisigned bytes too, not only the engine-asset cache G37 already re-verifies. **Secret-in-Actions-log lint (G56 sub-rule — the most direct leak of the three CI secrets into a PUBLIC repo's world-readable log, which GitHub's exact-string masking does NOT cover for a transformed/`set -x`-expanded secret):** a jq/yq-over-parsed-YAML check that the secret-bearing job's steps contain no `set -x`/`set -o xtrace`, echo/pipe no secret-named env var, and `::add-mask::`-wrap any secret-derived value; plus a release-tier post-run scan of the captured job log for the G2 secret SHAPES (minisign-key + `sk-ant-` + `MINISIGN_PASSWORD`), fail-closed on a match.
- **CI runner-host integrity** (G56 — content-independent, lives in `.github/`): the secret-bearing signing/release step is bound to an **ephemeral GitHub-hosted runner**, **never** the shared self-hosted VPS that runs the Lane-B untrusted-corpus/fuzz jobs (spec §6.1.4/§6.7.2); a workflow lint fails any secret-using job bound to a self-hosted label, asserts the corpus/fuzz job and the signing job declare disjoint hosts, and the **GitHub-hosted signing job uses `step-security/harden-runner` (BLOCK mode)** — asserted **only on GitHub-hosted jobs**, because harden-runner's free tier works only on GitHub-hosted runners (self-hosted needs Enterprise), so on the shared self-hosted VPS the egress enforcement is **G42/G42b + the VPS egress allowlist + an ephemeral/JIT low-priv runner**, not harden-runner. The `dependabot.yml` assertion covers **github-actions + cargo + npm + pip** ecosystems (not actions-only; pip for the gate-toolchain `requirements-ci.txt` layer — Semgrep et al. read CI secrets AND produce findings, r7). Implements security-concept principle 11. (Spec §6.7.2 is synced in the same change to bind the signing step to a hosted runner.)
- **GitHub branch-protection / required-status-checks config assertion** (G56a — content-independent, lives in `.github/` + a CI step): in the single-branch direct-to-`main` model the **only** thing that turns a red CI run into an actual block is GitHub repo config — invisible to the codebase and silently relaxable in the UI. A scheduled + per-push CI step queries the GitHub ruleset/branch-protection API for `main` and fails if the agreed required status checks are not all present-and-required, or `allow_force_pushes`/`allow_deletions`/admin-bypass is enabled; **plus three further invisible-config sub-checks** — GitHub native **secret-scanning + push-protection** enabled (`.security_and_analysis`); the repo **default workflow permissions = read-only** (`actions/permissions/workflow` → `default_workflow_permissions == "read"` + `can_approve_pull_request_reviews == false`, since a workflow omitting `permissions:` inherits this default, invisible to G50's per-file scan); **and inter-procedural TAINT depth for the T2/WebView surface as an OR-GATE** (`code-scanning/default-setup` → `state == "configured"` for `javascript-typescript`, OR the G29 rule (i) Semgrep `mode: taint` ruleset present with its planted-positive — r7; the depth the structural checks can't reach; **owner-decidable which ships, exactly one required, machine-enforced as an XOR by plan-lint check 21** so the sanctioned Semgrep path can never leave the required check permanently red — the prior hard CodeQL assert did); **and require-signed-commits on `main`** (`required_signatures` in the `main` ruleset — r7: the loop REUSES the P0.7-provisioned SSH signing key to sign its own loop commits, closing the integrity axis tag-only signing leaves open; an unsigned forged-trailer commit from a leaked `GITHUB_TOKEN` would otherwise feed the next release); **fail-soft only during the P0 bootstrap box, then hard.** (Spec §6.7.1 references G56a as the enforcement that replaces the removed "branch-protection + auto-merge" framing — synced in the same change.)
- **Release-tag (`v*`) trust assertion — the workflow half** (G56b — content-independent, lives in `.github/`): the secret-bearing minisign release job fires on a `v*` tag, which G56a does NOT cover (it guards only the `main` branch ref), so the release-workflow SKELETON's **first step asserts the tagged commit is an ancestor of `origin/main` AND main's required checks were green for that exact SHA** (`git merge-base --is-ancestor` + `gh api …/commits/<sha>/check-runs`) and aborts **before any secret is read** otherwise — so a `v*` tag on a commit that never passed L4 green (loop / compromised `GITHUB_TOKEN` / stale-forced tag) cannot mint a signed artifact; the `v*` tag is required to be a **signed annotated tag** and the first step runs **`git verify-tag` against the committed SSH allowed-signers file** (leg 3 — the loop signs its own `v*` tags; the allowed-signers file is provisioned in P0.7; distinct from spec §6.7.1's requested-not-required DCO commit sign-off). (The tag-protection ruleset POLICY half + the allowed-signers provisioning are homed in P0.7; legs 2/3 here are fail-closed always.)
- **Gate-toolchain pinning** (content-independent): every gate/SAST/SBOM/lint tool pinned by exact version AND verified by checksum/image digest at install (vendored binaries / digest-pinned containers over `cargo install`/`npm -g` from a live registry). The **fetch-and-verify mechanism is a self-tested control** — a script downloads each pinned tool, verifies its checksum/digest, and ships a **G24-style negative self-test** (a deliberately-wrong checksum MUST fail the install); `pip`-installed tools (Semgrep) use **`pip install --require-hashes -r requirements-ci.txt`** (a bare `pip install semgrep==X.Y.Z` verifies only the version string, not content), with a planted-positive that a hashless install fails the lint. **Gate-tool checksum-VALUE authenticity (r6 — the wrong-checksum self-test proves the verify mechanism, not that the committed hash was obtained authentically):** a committed gate-tool source allow-list (per-tool upstream origin), and at pin-establishment each recorded checksum is corroborated against the tool's OWN published signature (GitHub-release attestations / `cosign` / a maintainer-signed checksums file) OR ≥2 independent origins — applying the G37 acquisition-mode discipline to the gate tools (which both read the CI secrets and are what catches a real finding); a gate-tool checksum edit is a hard Co-Pilot escalation like an `engines.lock` SHA edit. A committed `rust-toolchain.toml` pins the exact stable channel + a **date-pinned** nightly (`nightly-YYYY-MM-DD`) for `cargo-fuzz` (G48), asserted not-floating. **`cargo-fuzz` invokes the date-pinned nightly EXPLICITLY (`cargo +nightly-YYYY-MM-DD fuzz`), never a bare `+nightly`** (cargo-fuzz internally calls `cargo +nightly …` using the active override, which does NOT necessarily honour `rust-toolchain.toml`'s channel field — so the date-pin is mechanically enforced only by the explicit channel, asserted by an actionlint/workflow-lint check that the fuzz job names the date-pinned channel and never a bare `+nightly`; verify against the pinned cargo-fuzz version). **The build toolchain that produces the shipped artifact is in the same trust boundary** ("everything that touches the bytes that get minisigned"): the CI base image/container is digest-pinned, the C/C++ toolchain and the Tauri CLI/bundler are version/digest-pinned; if engines are from-source, the source tarballs are hash-pinned + the build container digest-pinned (spec §3.8).
- **OSSF Scorecard informational corroboration** (G67 — reserved id, content-independent, `.github/`): an informational scheduled `ossf/scorecard-action` workflow (default-branch push + schedule) that independently corroborates the bespoke G50/G56/G56a assertions (branch-protection, pinned deps, dangerous workflows, token scope); a divergence is a Co-Pilot review item, never a required green check (a third-party scanner, not a deterministic in-repo gate).
- Defense-in-depth plane definitions + fail-open/closed policy wired as config.
- Fastpath / skip detectors (docs-only, check-off) + their self-tests (G10/G24) — the docs-only guard operates on the **range diff over all unpushed commits**, not the HEAD subject. **The L4 clean-checkout mirror runs EVERY custom gate-script's planted-positive self-test as a prelude on every run (not only when the script diffs, G24)** so a pinned-tool version bump that breaks a stable script's parsing is caught (the continuous-armed-canary pattern, complementing plan-lint check 16's "the self-tests exist").

**Home:** build-gates.md §0–§4 · 06-build-test-release (record the `actionlint`/`zizmor`/token-scope CI hardening in the spec per the living-doc rule).

**Goal:** the Opus/Sonnet review and the commit discipline are specified and

- **Reviewer rubric as a committed, drift-guarded artifact (r6 — the one piece of the G1 amplifier the loop can silently weaken):** the review PROMPT/RUBRIC the two reviewers receive (what they critique for — spec-conformance, the SPEC-CONTRADICTION-above-P0 class, severity ranking, divergence handling) is authored as **a fenced canonical block INSIDE `build-loop.md` (the home DECIDED in r7 — NOT a separate `docs/process/review-rubric.md` file)** so the "seven P0 docs" exit criterion stays exact (no eighth doc), added to the L(-1) security-critical-file set, with **plan-lint check 19** asserting the fenced block exists/non-empty, the emitted prompt is sourced from it verbatim, and it carries the canonical SPEC-CONTRADICTION-class + divergence-resolution phrases — so the autonomous loop cannot silently edit or under-specify its own review prompt and blunt the amplifier the spot-audit depends on.
- **Recorded reviewer-family decision (r6 — force-and-record the one open non-deterministic-control residual):** security-concept §4 honestly states Opus+Sonnet share lineage so "both GO, 0 findings" is a CORRELATED signal; the owner call (make one reviewer a different family, OR explicitly accept the correlated residual) **plus a concrete spot-audit cadence** (e.g. every Nth box / every phase boundary) is RECORDED verbatim in `build-loop.md`, with **plan-lint check 20** asserting the decision is present so the loop cannot silently ignore it.
- Review-trace commit trailer + its format gate (G12 — exact `^Dual-Review: opus=(GO|NOGO) sonnet=(GO|NOGO)$` at pre-push) **+ the findings-block-presence sub-check (r5): a `GO/GO` trailer on a non-trivial (non-check-off, non-`[!extern]`) diff must carry each reviewer's NON-EMPTY findings block — a both-GO + empty/absent block FAILS the gate, turning the §4 "auditable smell" into a machine invariant; raises the silent-`GO`-without-an-API-call cheat cost from one trailer line to a fabricated per-reviewer finding set (G1 stays a quality amplifier, not a security control — a fabricated block remains an auditable spot-audit target).**
- **Canonical check-off-commit shape (pinned in build-loop.md so the single shared G12/G54/fastpath skip-regex is authored against the EXACT string the loop emits — CLAUDE.md says docs German / code English but leaves the check-off commit-SUBJECT language unspecified, and a drift false-passes or false-fails valid check-off commits):** the check-off commit is a **double predicate** — subject matches **`chore(todo): .* (abgehakt|done)`** AND the diff is **markdown-only** (`git diff --name-only <commit>^..<commit> | grep -v '\.md$'` is empty). The loop knows exactly what to produce and G12/G54 know exactly what to recognise (closing the subject-only vs diff-shape ambiguity), with a **G54 planted-positive** (a check-off subject + md-only diff skips; a build-commit subject or a non-md diff does NOT).
- **Staged-diff sanity:** before `git commit`, `git diff --cached --stat` must match what the reviewers saw at GO; any post-GO file add/remove forces re-review.
- Commit conventions — Conventional-commit + body (spec-§ + box-id + findings + co-author trailer); solo-on-`main` rollback convention (`chore(scope): roll back — reason`, no `revert` type).
- **Definition-of-Done (the ConvertIA-adapted prior-project nine-point DoD, dropping RLS/audit/migrations):** (a) spec-§ referenced or marked tooling-only; (b) spec synced in the same commit; (c) tests at the highest sensible level green; (d) hard gates green with **no** `--no-verify`; (e) dual-review done + trailer; (f) inline `[Build-Session-Entscheidung]` at non-spec choice sites; (g) `engines.lock`+SBOM row if a new engine staged; (h) §0.11+§5 row if a new threat class introduced.
- **Build-loop soundness rules** (the autonomous-direct-to-main model depends on these): **(0) session-start sanity** — `git rev-parse --show-toplevel` MUST match the ConvertIA root (unconditional STOP on mismatch) + `git symbolic-ref --short HEAD` = `main`; correct remote, clean working tree, no half-committed/half-staged state before any box starts; **a `core.hooksPath`-not-redirected check** — `git config --get core.hooksPath` MUST be unset OR equal the lefthook-managed path (a redirection silently disables ALL local L1–L3 hooks without `--no-verify`, the un-named bypass the forbidden-bypass list now names; its verifying pre-push enforcement is G54, which resolves the effective hooks dir); **plus an out-of-band-tamper check** `git diff --exit-code HEAD -- scripts/ lefthook.yml .github/` (the two-plane principle does not cover a local out-of-band edit to a gate script); **plus a startup `[!]` dep-unlock scan BEFORE first-box selection (r6)** — run the full `[!]` dep-unlock scan (auto-flip a `[!]` box whose `unlocked-by:` dep is already `[x]` from a prior session) at startup, not only after each check-off, so a box another session already unblocked is selectable on this session's first box. **Security-critical-file change-control (owner-decidable — the loop authors its own cage):** the out-of-band-tamper check + the gate-currency fetch protect only against an EXTERNAL edit landing mid-session; neither protects against the loop ITSELF mis-editing its cage (the gate scripts, gate self-tests, `lefthook.yml`, `deny.toml`, `.gitleaks.toml` **+ its allowlist/baseline**, the cargo-vet `supply-chain/config.toml` exemption set, the G56b SSH allowed-signers file, capabilities, `engines.lock`, **`.gitattributes` + `.lfsconfig` (which govern LFS resolution — G24a, r7)**, the reviewer rubric, the security docs) and self-attesting the dual review. The base-case meta-checks (the plan-lint golden-fixture-must-exit-1 + the G24 planted-positive prelude) prove the gates still DETECT a planted violation each run, which is the deterministic backstop. **Owner-decidable L(-1) escalation (not a default mandate, because a hard pre-push Co-Pilot-ack changes the autonomous model):** define a SECURITY-CRITICAL FILE SET and require any commit touching it to be **Co-Pilot-acked before push** (app code stays autonomous; the cage does not) — recorded as an owner decision, since it is the one structural answer to "what if the thing running the gates is wrong about security in a way both same-lineage reviewers share". **(1) session-start CI health (+ per-box, r6)** — query the last Lane-A run on `main`, **filtered to `--workflow <main-lane-A workflow> --event push --branch main`** (without the filter a scheduled Scorecard/G56a run or a tag/release run false-passes or false-stops the loop); `gh run list --workflow <lane-A> --event push --branch main --limit 1 --json status,conclusion,headSha`; STOP+escalate on failure/cancelled, proceed on success/pending/queued, fail-open (warn+continue) if the API is unreachable. **Fire this check at EVERY box start, not only session start** (1 cheap `gh` call), to close the mid-session red-`main` gap between step-2a (after a push) and the next box — adopting RMA's per-box CI-health discipline. **(1a) session-start gate-currency** — `git fetch origin main && git diff --name-only HEAD origin/main`: if any gate file (`lefthook.yml`, `scripts/**`, `.github/**`, `deny.toml`, `.gitleaks.toml`, `.npmrc`, lockfiles) drifted on origin (a Co-Pilot/owner gate bugfix mid-session) STOP+escalate so the loop does not continue on stale gate scripts. **Its verifying pre-push enforcement is the G54 push-from-stale-base guard** (`git merge-base HEAD origin/main == origin/main` — HEAD descends from the current remote tip), so a push from a stale base that would overwrite an already-pushed gate fix is caught at push time, not after. **(2) push-exit-code wait** — the push exit code MUST be reliably observed. **The agent tool environment does not propagate a subprocess exit code the way a plain bash script does: `| tee` masks the hook's non-zero exit, and a naive `$?` can capture the tool-call's own success rather than git-push's.** The proven fix (a prior-project pattern, ported) is a **background-push + marker-file + a synchronous foreground until-loop polling the marker file**: `{ git push origin main; echo "EXIT=$?" >> "$DONE"; } > "$LOG" 2>&1` then a **foreground** poll of `$DONE` for the `EXIT=` line, reading the captured code from it (NOT a bare `$?`). **`run_in_background` is FORBIDDEN for the wait-loop** (the loop must block synchronously on the marker), as are `| tee` and `pgrep`-polling of the push process. The **same marker-file capture applies to the check-off (hak) commit push — it is NOT exempt**. (Verify the capture mechanism against the real tool behaviour before relying on a bare `$?`; this is the prior project's hard-won fix for exactly this model. `build-loop.md` carries the concrete marker-file recipe.) On non-zero do NOT proceed to the check-off commit or next box (fix, re-stage, re-review, retry; escalate after **3 consecutive push failures**); never start a new box while a push is unresolved. **(2a) watch own CI run** — because the loop pushes **direct to `main`**, its own push's CI failure IS a red `main`: capture the run-ID of the just-triggered run **SHA-anchored** — `git rev-parse HEAD`, then poll `gh run list --branch main --json databaseId,headSha,status` filtering `headSha == <commit-sha>` (NOT `--limit 1` "the most recent run", which a fast second push — the hak/check-off commit — can STEAL, making the loop watch the wrong run), then **`gh run watch --exit-status <run-id>`** (`--exit-status` is **MANDATORY** — without it `gh run watch` exits 0 even on a failed run, a no-op guard; document this so a future reader cannot silently drop it); a non-zero exit ⇒ same STOP+escalate as session-start. A **`cancelled` conclusion caused by the G56 concurrency cancel-in-progress** is reconciled by the **successor-exists check, NOT a "newer run for the same SHA"** (the G56 cancel produces a newer run for a *later* SHA, not the same SHA — the same-SHA-newer-run scenario does not occur in normal operation): `gh run list --branch main --limit 3 --json databaseId,headSha,conclusion,status`; **if** the cancel has a successor (a higher-`databaseId` run for a later SHA in-progress/succeeded) the cancel was the expected G56 cancel and the loop waits on the successor; **if NO successor exists, the cancel is anomalous → STOP+escalate**. **G42/G42b egress-window protection — ONE mandatory rule (r6, collapsed from a misleading "either/or"):** the two earlier options are NOT equivalent — `cancel-in-progress: false` prevents a mid-window KILL but does NOT stop step-2a from marking a *cancelled* box-commit run done when the later hak-push run is its successor; only **deferring the check-off (hak) push until the box-commit run completes green** closes that. So the MANDATORY rule is: **the loop MUST NOT push the check-off commit until the box-commit run completes green** (`gh run watch --exit-status <box-run-id>` returns 0). `cancel-in-progress: false` on the G42/G42b job group is kept as a **belt-and-suspenders G56 assertion, NOT an alternative**. step-2a additionally **verifies G42/G42b specifically PASSED**, not just that some newer run exists. A **G56 workflow-lint sub-rule** asserts no workflow including the G42/G42b job group sets `cancel-in-progress: true` on that group. **(3) next-box selection** — lowest-phase/lowest-box first; `[!extern]`/`[!]`-blocked boxes skipped+reported; after each check-off, scan for `[!]` boxes the just-completed box unblocked (an `unlocked-by: <box-id>` marker in `_format.md` flips `[!]`→`[ ]` when its dep completes) and unlock them; on zero open boxes report convergence and stop (never loop forever); escalate on a genuine all-blocked deadlock.
- **Dual-review availability soundness (sibling to the 3-push-failures rule):** the two reviewer **model IDs are pinned** (exact IDs, like every other tool, so a deprecation surfaces as an escalation not a silent skip); on a reviewer error/timeout/rate-limit/5xx, retry with backoff a bounded N, then **HARD-STOP+escalate** — **NEVER** auto-emit a `GO` trailer with fewer than **two live** reviews, never silently single-review (G12 only checks the trailer is well-formed). The same-vendor correlated-blind-spot residual is named in security-concept §4.
- **Gate-quarantine procedure (a provably-misfiring required gate in a no-bypass, single-branch model):** a required gate that fails CLOSED on a **false positive** (a tool regression on a version bump, a Semgrep/zizmor/actionlint misfire) would otherwise WEDGE the autonomous loop with no sanctioned escape that isn't a forbidden `--no-verify`. The only sanctioned unblock is a **COMMITTED, dual-reviewed change** that fixes or **narrowly scopes or temporarily suppresses the specific gate check** (with a tracked box-id to restore it) — so the fix goes through the same gates and the gate is never silently skipped; a provably-misfiring gate is otherwise a hard-stop+escalate to Co-Pilot. **Bootstrap exception for a self-referential deadlock (a gate that fails CLOSED on its OWN fix commit):** a gate that misfires on an L1 false positive blocks even the fix commit (the misfiring L1 hook fires on it), and the same circularity hits **plan-lint** (G7/G20, which enforces its own doc-consistency invariants checks 8/14/16/17 — a plan-lint parser bug can't be fixed because the fix must pass plan-lint). For exactly this chicken-and-egg path the sanctioned escape is a **COMMITTED, narrowly-scoped edit to `lefthook.yml`** that comments out ONLY that one gate command (a `lefthook.yml` edit passes a DIFFERENT glob than the misfiring gate, so it is not blocked by it) — **never `--no-verify`** — with a paired `[Build-Session-Entscheidung]` tag and a restore box-id; the gate is re-enabled in the immediately following box. The **plan-lint circularity case is named specifically**: if plan-lint itself is the misfiring gate, the same `lefthook.yml`-comment-out-then-restore path applies (and the G65/G24 base-case meta-check — a golden fixture repo-state that MUST make plan-lint exit 1 — proves plan-lint still detects a planted violation after the fix). **Per-finding suppression ledger (finer-grained than the gate-quarantine sledgehammer, for G29 Semgrep / G17b–G18 triage):** a single legitimate Semgrep/advisory false-positive should NOT force a gate-wide suppression or a hard-stop — so a committed **suppression ledger** records each suppressed finding with a **content-derived fingerprint + box-id + Dual-Review note**, and the **fingerprint ROTATES** (re-requires justification) when the surrounding code changes (so a suppression cannot silently outlive the code it was reasoned about). The gate-quarantine scope/suppress escape stays for a tool-level misfire that has no per-finding fingerprint.
- **Hard-stop / token-Notbremse numbers (decided here, the ConvertIA baselines):** **soft-stop ~8 boxes** (pause + summarize for the owner), **hard-stop ~12 boxes** in one session, **phase-change hard-stop** after ≥1 committed box of a new phase (re-orient at a phase boundary) — **this applies to TOP-LEVEL phases (P0→P1…), NOT the P0.1–P0.7 clusters** (P0 is one phase; firing it per cluster would mean 6 needless interruptions in the most important phase), so P0 instead takes a **cluster-boundary SOFT-stop that fires AT a P0.x cluster boundary once ≥5 boxes have been COMMITTED since the last soft-stop** (a running counter that RESETS at each soft-stop, not a per-cluster reset and not a fire-at-count-regardless-of-boundary — it pauses at the next cluster boundary after the counter crosses 5; plan-lint check 15 verifies this exact semantics). **The counter increments ONLY on a COMMITTED box — `[!extern]`/`[!]`-skipped boxes do NOT count (r6, stated explicitly so a soft-stop cannot fire spuriously after 5 skips, consistent with the phase-change hard-stop's count-0 exemption).** The phase-change hard-stop does **NOT** fire if the counter is **0** (all phase boxes were `[!extern]`/`[!]`-skipped); **3 consecutive gate-red pushes = hard-stop+escalate**. (Adapted from the RMA model; digits are the v1 baseline, tunable.)
- **Opt-in box-batching (RMA refinement):** at most **3 sister-boxes** in the same cluster, no sub-boxes / no cross-deps, trivial repetition only (e.g. the structurally-identical per-engine rows in P0.7) ⇒ **1 build-commit + 1 dual-review over the combined diff + 1 check-off**, counter `+= N`; without this rule the loop either wastes dual-review cycles on identical rows or batches implicitly with no rule. **Dual-review fires once per TOP-box** (over the combined sub-box diff), not per sub-box.
- **Spec-contradiction hard-stop:** a **spec-internal contradiction** (two §§ disagree) is an **unconditional hard-stop+escalate regardless of P-severity** — the Build-Loop is downstream of the spec and cannot pick a side. The dual-review prompt instructs **both reviewers to flag a spec §-cross-reference inconsistency as a SPEC-CONTRADICTION finding class above P0** (unconditional hard-stop+escalate, **never a working-tree fix**).

- **Per-box status-line format** (the prior-project *output-discipline* convention — distinct from the Definition-of-Done): one line per box — `P0.X done — <summary>, gates green, SHA <short>, Review: P0=N P1=N P2=N P3=N` + a separate `Co-Pilot: N item(s)` block for P1+ findings — so session progress + escalation signals are scannable.

- **Crash-recovery procedure (named build-loop.md deliverable — plan-lint check 18 asserts a canonical phrase for it appears in build-loop.md, so it cannot stay unwritten):** **(a)** a partial staged state → `git reset HEAD` + re-read the box (no half-staged commit); **(b)** committed-but-CI-red → a NEW commit fixing it, **never amend a pushed commit**; **(c)** pushed-but-not-checked-off → the normal open-box scan catches it (the check-off is idempotent on retry); **(d)** push is idempotent on retry. So a session crash mid-box is recoverable without manual surgery.
- **Dual-review divergence-resolution rule (named build-loop.md deliverable — plan-lint check 18 asserts a canonical phrase for it appears in build-loop.md):** a **P0/P1 GO-vs-NOGO divergence is treated as NOGO (the stricter reviewer wins)**; a **P2/P3 divergence is resolved by the loop with a recorded `[Build-Session-Entscheidung]` rationale** — UNLESS it is a spec-contradiction, which is the unconditional hard-stop+escalate above (the loop is downstream of the spec and cannot pick a side).

- **Vuln-response runbook** (`docs/process/vuln-response.md`) — because the app ships known-vulnerable-class C/C++ decoders against untrusted files with **no auto-update**, the only path a security fix reaches a user is a new full release: advisory (G17b/upstream) → Build-Loop escalates to Co-Pilot → bump the `engines.lock` pin → re-run the §6.5 reliability gate → new release; matches the `SECURITY.md` "no SLA, best-effort" posture with an actual triage process. **Severity threshold (stated in `SECURITY.md`):** a CVE with **CVSS ≥ 7 on an engine code path ConvertIA actively exercises for a §04 format** → MUST escalate + block the next release until bumped or triaged not-exercised (G17b release-tier rule), so a CVSS≥9 PoC in an actively-exercised FFmpeg/poppler path can't sit shipped for weeks. **Beyond engine CVEs, the runbook also covers:** (a) **own-code (non-engine) vulns** + the coordinated-disclosure intake→embargo→fix→release loop (via the private-advisory template, §6.8); (b) a **signing-key compromise/loss path** — the human-readable "this key is retired/compromised" commit IS the revocation channel for an offline no-phone-home app (an authored artifact), plus an offline encrypted key+passphrase backup/custody note (the §6.2.3 rotation policy is the silent-swap defence; this is the incident path it doesn't cover); (c) **a confirmed high-severity engine vuln with NO upstream fix yet (r6 — the DOMINANT real-world case for a no-auto-update offline app shipping legacy C/C++ decoders, where a CVE often predates a patched release by weeks/months, so "bump the pin" assumes a fix that doesn't exist):** ranked options — **disable the specific decoder** via the G38 `ffmpeg-allowed-decoders.lock` allow-list (the surface-reduction lever already built) / **drop the affected format path** and ship; **publish a documented mitigation / format-restriction**; **escalate the disable-vs-ship call to Co-Pilot** — tied to a `SECURITY.md` known-issues line (the only channel an offline app has: release notes / the user-initiated Releases link), so an actively-exercised CVSS≥9 PoC with no patch has a defined response instead of leaving users on a stale build for years.
