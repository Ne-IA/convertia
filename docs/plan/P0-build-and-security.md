# P0 — Build & Security System

> **The foundation before the foundation.** P0 establishes the guardrail system
> every later phase (P1–P11) runs under: the security concept, the full gate
> system, the build-loop, the dual review, and the test methodology. It is built
> **before** P1 writes any app code, so P1's very first commit already runs through
> the loop + the gates.
>
> Derives from [security-concept.md](../security/security-concept.md) +
> [build-gates.md](../security/build-gates.md). Index: [plan/README.md](README.md).
>
> **This file currently holds the cluster structure + box headings only** — the
> atomic `[ ]` steps are added in the fill pass, after this structure is reviewed
> and signed off. No `[ ]` boxes are filled yet.

## Boundaries (decided, see plan/README.md)

- **P0 ↔ P1:** P0 builds the gate *system* + content-independent gates + the
  framework/wiring-points for language gates; **P1** wires the language-specific
  gates (clippy/tsc/cargo-deny/…) into the P0 framework as it scaffolds, and
  authors the general governance docs.
- **P0 ↔ P10:** P0 *defines* the release-/supply-chain-gate policy; **P10** builds
  the release pipeline. No double-build.
- **Content-independence rule:** a P0 box may only build what can exist without app
  code. Gates that need Rust/TS to act on are *defined + wired-pointed* here and
  *activated* by the producing phase (annotate `→ activated in P1`).

## P0.2 — Gate orchestration framework

**Goal:** the two enforcement planes exist as empty-but-wired harnesses that later
gates plug into.

**Box areas:**
- Git-hook manager (`lefthook`) set up (pre-commit / pre-push / commit-msg), `parallel`, perf budgets; the **hooks-installed assertion** (G54 — `lefthook install` mandatory after clone, no local protection without it).
- GitHub Actions CI skeleton (L4) — clean-checkout job matrix placeholders (Win/macOS/Linux).
- Release workflow skeleton (L5) — tag-triggered, empty gate slots.
- **CI supply-chain hardening** (content-independent — lives in `.github/`): every workflow declares top-level `permissions: contents: read`, elevated per-job only where needed (the release job touching `MINISIGN_SECRET_KEY`/`MINISIGN_PASSWORD` gets `contents: write` ONLY); every third-party action pinned by **full 40-char commit SHA** (not a tag), kept current via a `dependabot.yml` github-actions entry (its presence asserted by G56); `actionlint` (G49, L1) + `zizmor` (G50, L4) workflow lint; the secret-bearing release job **never runs on a fork pull-request**; per-PUSH-workflow `concurrency: {group, cancel-in-progress: true}` (never on the release/tag workflow) + explicit `timeout-minutes` per job. **Cache-hygiene lint (G56 sub-rule — build-cache poisoning defence, generalising G37's "an Actions cache is not integrity-protected"):** the secret-bearing release/signing job restores no writable shared cache (cache disabled OR read-only / lockfile+toolchain-derived key); fork-PR workflows share no cache-key namespace with release/`main` (no `restore-keys` prefix bleed); the signed artifact builds `--locked`/`--frozen-lockfile` from a cache-cold or integrity-verified tree — the Cargo/pnpm/sccache/gate-tool/fuzz-corpus caches feed the minisigned bytes too, not only the engine-asset cache G37 already re-verifies.
- **CI runner-host integrity** (G56 — content-independent, lives in `.github/`): the secret-bearing signing/release step is bound to an **ephemeral GitHub-hosted runner**, **never** the shared self-hosted VPS that runs the Lane-B untrusted-corpus/fuzz jobs (spec §6.1.4/§6.7.2); a workflow lint fails any secret-using job bound to a self-hosted label, asserts the corpus/fuzz job and the signing job declare disjoint hosts, and the **GitHub-hosted signing job uses `step-security/harden-runner` (BLOCK mode)** — asserted **only on GitHub-hosted jobs**, because harden-runner's free tier works only on GitHub-hosted runners (self-hosted needs Enterprise), so on the shared self-hosted VPS the egress enforcement is **G42/G42b + the VPS egress allowlist + an ephemeral/JIT low-priv runner**, not harden-runner. The `dependabot.yml` assertion covers **github-actions + cargo + npm** ecosystems (not actions-only). Implements security-concept principle 11. (Spec §6.7.2 is synced in the same change to bind the signing step to a hosted runner.)
- **GitHub branch-protection / required-status-checks config assertion** (G56a — content-independent, lives in `.github/` + a CI step): in the single-branch direct-to-`main` model the **only** thing that turns a red CI run into an actual block is GitHub repo config — invisible to the codebase and silently relaxable in the UI. A scheduled + per-push CI step queries the GitHub ruleset/branch-protection API for `main` and fails if the agreed required status checks are not all present-and-required, or `allow_force_pushes`/`allow_deletions`/admin-bypass is enabled; **fail-soft only during the P0 bootstrap box, then hard.** (Spec §6.7.1 references G56a as the enforcement that replaces the removed "branch-protection + auto-merge" framing — synced in the same change.)
- **Gate-toolchain pinning** (content-independent): every gate/SAST/SBOM/lint tool pinned by exact version AND verified by checksum/image digest at install (vendored binaries / digest-pinned containers over `cargo install`/`npm -g` from a live registry). The **fetch-and-verify mechanism is a self-tested control** — a script downloads each pinned tool, verifies its checksum/digest, and ships a **G24-style negative self-test** (a deliberately-wrong checksum MUST fail the install); `pip`-installed tools (Semgrep) use **`pip install --require-hashes -r requirements-ci.txt`** (a bare `pip install semgrep==X.Y.Z` verifies only the version string, not content), with a planted-positive that a hashless install fails the lint. A committed `rust-toolchain.toml` pins the exact stable channel + a **date-pinned** nightly (`nightly-YYYY-MM-DD`) for `cargo-fuzz` (G48), asserted not-floating. **The build toolchain that produces the shipped artifact is in the same trust boundary** ("everything that touches the bytes that get minisigned"): the CI base image/container is digest-pinned, the C/C++ toolchain and the Tauri CLI/bundler are version/digest-pinned; if engines are from-source, the source tarballs are hash-pinned + the build container digest-pinned (spec §3.8).
- Defense-in-depth plane definitions + fail-open/closed policy wired as config.
- Fastpath / skip detectors (docs-only, check-off) + their self-tests (G10/G24) — the docs-only guard operates on the **range diff over all unpushed commits**, not the HEAD subject. **The L4 clean-checkout mirror runs EVERY custom gate-script's planted-positive self-test as a prelude on every run (not only when the script diffs, G24)** so a pinned-tool version bump that breaks a stable script's parsing is caught (the continuous-armed-canary pattern, complementing plan-lint check 16's "the self-tests exist").

**Home:** build-gates.md §0–§4 · 06-build-test-release (record the `actionlint`/`zizmor`/token-scope CI hardening in the spec per the living-doc rule).

## P0.5 — Test methodology & harness conventions

**Goal:** *how we write tests* is defined and the cross-cutting test invariants
have a home.

**Box areas:**
- Test-levels doctrine: unit · integration (real files, **never mock the thing under test**) · property · fuzz · E2E · a11y (the §6.4.6a jsdom/`vitest-axe` ARIA leg = G33a — **`→ activated in P1`** with the rendered React tree, carrying the same bootstrap annotation as G27/G47/G42; vs the Lane-B `@axe-core/webdriverio` contrast leg = G33b).
- Corpus / fixture conventions — single-source helper, auto-discovery, no inline duplication; **explicit decompression-bomb corpus FIXTURES** (svgz bomb, ZIP-bomb-in-OPC DOCX, deeply-nested PDF flate stream) so the §6.4.2 bomb case is backed by files, not just a property concept; **two deterministic bound-firing fixtures** (a gzip exactly 101× compressed → bounded `Err`; a svgz exactly `MAX_SVGZ_SNIFF + 1` → sniff stops at the limit) so a removed cap is caught structurally, not fuzzer-hoped (G48/G16). **Corpus/crash-fixture integrity (G24a):** a committed **SHA-256 manifest of every tracked corpus/crash fixture** (and the LFS-resolved `corpus-large` objects) verified in CI **before** the corpus runs, plus `git lfs fsck` on the Lane-B leg — these untrusted-input files are fed to the highest-privilege C/C++ decoders, so a poisoned/swapped fixture or a redirected LFS pointer must surface as a manifest diff (a dual-review item), mirroring the `engines.lock` SHA discipline.
- **Source-unchanged + output-validity invariant defined (G32 — replaces the vacuous "A→B→A byte-stable / within tolerance"):** (a) SOURCE-UNCHANGED — `sha256` of every corpus source unchanged before/after (the no-harm proof); (b) OUTPUT-VALIDITY — produced output passes a REAL per-format structural check (ffprobe decodable+codec, `vipsheader` decode+dims, poppler opens, `unzip`+`[Content_Types].xml`, CSV/TSV via a **real RFC-4180 reader (the `csv` crate)** + CSV-injection literal-preservation, **NOT bare field-count parity** — which passes on mis-quoted/embedded-newline output that is unparseable; see line below + G31/G32), not magic-sniff; the literal byte-stable check is scoped to the small truly-invertible set (e.g. PNG→BMP→PNG) with the covered pairs documented.
- Property-test conventions — **language-split, not either/or:** Rust = **`proptest`** (macro-based shrinking, no manual `Shrink` impls — satisfies "shrinking mandatory"); TS = **`fast-check`** (custom `Arbitrary` must delegate to built-in shrinkers — ban `fc.gen()` without a shrink wrapper); **determinism:** pinned CI seed, a case-count floor above the thin default 256, a property failure NEVER retried to pass. The coverage-guided **`cargo-fuzz`** harness is the **separate** in-core target set — `crate::detect` + `crate::fs_guard::resolve_identity`/`is_safe_output` (untrusted paths: null bytes, overlong UTF-8, symlink chains, `..`) + the in-core CSV/TSV engine (gigabyte quoted fields, recursive quoting, NUL) (G48), with the XML peek's DTD/external-entity resolution disabled by construction; **every libFuzzer crash is minimized + committed under `fuzz/corpus/`+`fuzz/crashes/` and the deterministic replay runs the committed crash set on EVERY platform incl. Windows.**
- Flaky-test policy — retry infra/timeout only, E2E-only auto-retry, determinism engineered (pinned locale/timezone, animations off).
- "Build fully, no skeleton/stub" rule wired to the deferral gate (G8).
- Coverage thresholds (per-domain) + shard-merge determinism.
- **Cross-cutting security-test homes (defined here, activated by their phase):** the §7.5 **log-redaction** property gate (feed a secret-looking path stem through the logger, assert absent), the §2.14.1 **temp ownership + mode-bits** assertion (`0o700` scratch root / `0o600` `.part` publish-temp), the **T10 adversarial resource-budget** gate (oversized-render SVG, over-duration to-GIF, over-cardinality batch → fail-clearly, batch continues, no handle/RAM exhaustion), the **T9b corpus sentinels** (`.docm`/`.xlsm`/`.pptm` AutoOpen-macro canary NOT created; `WEBSERVICE()` `.xlsx` no-egress/no-out-of-input), the **T7 INPUT-side symlink/junction** case (resolved-real-path handed to the engine), the **Windows AV-retry** fault-injection (§2.1.2), the **privilege-drop-tier-applied** regression assertion (§2.12.3 silent-degrade can't disable on every run unnoticed) **+ the privilege-drop-tier RATCHET (G64 — record the achieved tier per platform in a tracked `privilege-drop-coverage.toml`, decrease-guarded like the coverage floor, so a NET regression of the most consequential runtime security property fails/escalates; owner-decidable required-vs-informational like G17b)**, the **§2.12.3 memory-cap kill** assertion (an engine exceeding its Job-Object/`RLIMIT` memory budget mid-conversion is killed to a clean `Failed(TooBig|EngineHang)`, batch continues, host RSS returns to baseline), and the **process-group/Job-Object reap** assertion (no orphan/zombie).
- **Atomicity-under-interruption** (§6.4.2): the kill is injected **specifically in the post-`sync_all()`-pre-`rename` window** (a `#[cfg(test)]` fence in `crate::fs_guard::atomic_publish`, all 3 OS) to exercise the §2.1.3 two-state invariant at the critical boundary.
- **Scoped mutation-testing gate** (`cargo-mutants` over `crate::fs_guard` + `crate::detect` + `crate::outcome` — the no-harm/atomicity/no-misroute kernel) as a release-tier informational-then-ratcheted gate (line coverage proves a line ran, not that a test would CATCH a regression there); **owner-decidable** required-vs-informational like G17b. **Activation criteria defined here (so "informational-then-ratcheted" is actionable):** the initial informational run outputs a **survived-mutant report per kernel crate**; the ratchet is a tracked `max_survived_mutants.toml` per crate, initialised at first-run count, **decrease-only** (same pattern as the coverage floor); the owner flips informational→required when the count reaches **0** for `crate::fs_guard` + `crate::detect` (the two highest-consequence surfaces).
- **Output-validity per-format reader conventions (G31/G32):** name the REAL structural readers (ffprobe decodable+codec; `vipsheader` decode+dims; poppler opens; `unzip`+`[Content_Types].xml`; CSV/TSV via a **real RFC-4180 reader (the `csv` crate)** + a CSV-injection-literal-preservation assertion, NOT a bare field-count) and the **cross-library decode validation** for the headline formats (AVIF/HEIC re-validated with `ffprobe` — a different decoder family from the producing libheif/libvips — animated WEBP via `ffprobe`, NOT `dwebp` which cannot decode animated WebP; skip a pair's cross-check if the staged FFmpeg build lacks the decoder rather than false-pass).

## P0.6 — Dual review (holy grail) & commit protocol

**Goal:** the Opus/Sonnet review and the commit discipline are specified and
enforceable.

**Box areas:**
- Dual-review protocol (G1) — staged-diff input, P0–P3 severity, converge/diverge, fix-before-push (no fix-push cycle), skip conditions; **the dual review is a quality amplifier, NOT a security control** (only the deterministic gates — every `Gnn` except G1 — are security controls).
- Review-trace commit trailer + its format gate (G12 — exact `^Dual-Review: opus=(GO|NOGO) sonnet=(GO|NOGO)$` at pre-push).
- **Canonical check-off-commit shape (pinned in build-loop.md so the single shared G12/G54/fastpath skip-regex is authored against the EXACT string the loop emits — CLAUDE.md says docs German / code English but leaves the check-off commit-SUBJECT language unspecified, and a drift false-passes or false-fails valid check-off commits):** the check-off commit is a **double predicate** — subject matches **`chore(todo): .* (abgehakt|done)`** AND the diff is **markdown-only** (`git diff --name-only <commit>^..<commit> | grep -v '\.md$'` is empty). The loop knows exactly what to produce and G12/G54 know exactly what to recognise (closing the subject-only vs diff-shape ambiguity), with a **G54 planted-positive** (a check-off subject + md-only diff skips; a build-commit subject or a non-md diff does NOT).
- **Staged-diff sanity:** before `git commit`, `git diff --cached --stat` must match what the reviewers saw at GO; any post-GO file add/remove forces re-review.
- Commit conventions — Conventional-commit + body (spec-§ + box-id + findings + co-author trailer); solo-on-`main` rollback convention (`chore(scope): roll back — reason`, no `revert` type).
- **Definition-of-Done (the ConvertIA-adapted RMACLAUDE §11, dropping RLS/audit/migrations):** (a) spec-§ referenced or marked tooling-only; (b) spec synced in the same commit; (c) tests at the highest sensible level green; (d) hard gates green with **no** `--no-verify`; (e) dual-review done + trailer; (f) inline `[Build-Session-Entscheidung]` at non-spec choice sites; (g) `engines.lock`+SBOM row if a new engine staged; (h) §0.11+§5 row if a new threat class introduced.
- **Build-loop soundness rules** (the autonomous-direct-to-main model depends on these): **(0) session-start sanity** — `git rev-parse --show-toplevel` MUST match the ConvertIA root (unconditional STOP on mismatch) + `git symbolic-ref --short HEAD` = `main`; correct remote, clean working tree, no half-committed/half-staged state before any box starts; **plus an out-of-band-tamper check** `git diff --exit-code HEAD -- scripts/ lefthook.yml .github/` (the two-plane principle does not cover a local out-of-band edit to a gate script). **(1) session-start CI health** — query the last Actions run on `main` (`gh run list --branch main --limit 1 --json status,conclusion`); STOP+escalate on failure/cancelled, proceed on success/pending/queued, fail-open (warn+continue) if the API is unreachable. **(1a) session-start gate-currency** — `git fetch origin main && git diff --name-only HEAD origin/main`: if any gate file (`lefthook.yml`, `scripts/**`, `.github/**`, `deny.toml`, `.gitleaks.toml`, `.npmrc`, lockfiles) drifted on origin (a Co-Pilot/owner gate bugfix mid-session) STOP+escalate so the loop does not continue on stale gate scripts. **Its verifying pre-push enforcement is the G54 push-from-stale-base guard** (`git merge-base HEAD origin/main == origin/main` — HEAD descends from the current remote tip), so a push from a stale base that would overwrite an already-pushed gate fix is caught at push time, not after. **(2) push-exit-code wait** — the push is a **foreground `git push` with the exit code captured directly**: `git push origin main; push_exit=$?; if [ $push_exit -ne 0 ]; then escalate; fi` — **explicitly forbid `| tee`** (masks the hook's non-zero exit, the RMA Schritt-9 quirk), `run_in_background`, and `pgrep`-polling; the **same foreground exit-code rule applies to the check-off (hak) commit push — it is NOT exempt**. On non-zero do NOT proceed to the check-off commit or next box (fix, re-stage, re-review, retry; escalate after **3 consecutive push failures**); never start a new box while a push is unresolved. **(2a) watch own CI run** — because the loop pushes **direct to `main`**, its own push's CI failure IS a red `main`: capture the run-ID of the just-triggered run (poll `gh run list --branch main --limit 1 --json databaseId,status` for the new run), then **`gh run watch --exit-status <run-id>`** (`--exit-status` is **MANDATORY** — without it `gh run watch` exits 0 even on a failed run, a no-op guard; document this so a future reader cannot silently drop it); a non-zero exit ⇒ same STOP+escalate as session-start. A **`cancelled` conclusion caused by the G56 concurrency cancel-in-progress** is reconciled by the **successor-exists check, NOT a "newer run for the same SHA"** (the G56 cancel produces a newer run for a *later* SHA, not the same SHA — the same-SHA-newer-run scenario does not occur in normal operation): `gh run list --branch main --limit 3 --json databaseId,headSha,conclusion,status`; **if** the cancel has a successor (a higher-`databaseId` run for a later SHA in-progress/succeeded) the cancel was the expected G56 cancel and the loop waits on the successor; **if NO successor exists, the cancel is anomalous → STOP+escalate**. **G42/G42b egress-window protection (a long-running egress-deny + canary leg must NOT be killed mid-flight by the G56 cancel):** either **defer the check-off push until the box-commit CI completes green** (`gh run watch --exit-status <box-run-id>` returns 0), OR put the G42/G42b Lane-B job in a job group with **`cancel-in-progress: false`** (never cancel an egress-deny window mid-flight — a kill mid-window yields a `cancelled` conclusion that step-2a would otherwise treat as expected, silently skipping G42/G42b for that push with no gate noticing), and make step-2a **verify G42/G42b specifically PASSED**, not just that some newer run exists. A **G56 workflow-lint sub-rule** asserts no workflow including the G42/G42b job group sets `cancel-in-progress: true` on that group. **(3) next-box selection** — lowest-phase/lowest-box first; `[!extern]`/`[!]`-blocked boxes skipped+reported; after each check-off, scan for `[!]` boxes the just-completed box unblocked (an `unlocked-by: <box-id>` marker in `_format.md` flips `[!]`→`[ ]` when its dep completes) and unlock them; on zero open boxes report convergence and stop (never loop forever); escalate on a genuine all-blocked deadlock.
- **Dual-review availability soundness (sibling to the 3-push-failures rule):** the two reviewer **model IDs are pinned** (exact IDs, like every other tool, so a deprecation surfaces as an escalation not a silent skip); on a reviewer error/timeout/rate-limit/5xx, retry with backoff a bounded N, then **HARD-STOP+escalate** — **NEVER** auto-emit a `GO` trailer with fewer than **two live** reviews, never silently single-review (G12 only checks the trailer is well-formed). The same-vendor correlated-blind-spot residual is named in security-concept §4.
- **Gate-quarantine procedure (a provably-misfiring required gate in a no-bypass, single-branch model):** a required gate that fails CLOSED on a **false positive** (a tool regression on a version bump, a Semgrep/zizmor/actionlint misfire) would otherwise WEDGE the autonomous loop with no sanctioned escape that isn't a forbidden `--no-verify`. The only sanctioned unblock is a **COMMITTED, dual-reviewed change** that fixes or **narrowly+temporarily downgrades the specific gate** (with a tracked box-id to restore it) — so the fix goes through the same gates and the gate is never silently skipped; a provably-misfiring gate is otherwise a hard-stop+escalate to Co-Pilot. **Per-finding suppression ledger (finer-grained than the gate-quarantine sledgehammer, for G29 Semgrep / G17b–G18 triage):** a single legitimate Semgrep/advisory false-positive should NOT force a gate-wide downgrade or a hard-stop — so a committed **suppression ledger** records each suppressed finding with a **content-derived fingerprint + box-id + Dual-Review note**, and the **fingerprint ROTATES** (re-requires justification) when the surrounding code changes (so a suppression cannot silently outlive the code it was reasoned about). The gate-quarantine downgrade stays the escape for a tool-level misfire that has no per-finding fingerprint.
- **Hard-stop / token-Notbremse numbers (decided here, the ConvertIA baselines):** **soft-stop ~8 boxes** (pause + summarize for the owner), **hard-stop ~12 boxes** in one session, **phase-change hard-stop** after ≥1 committed box of a new phase (re-orient at a phase boundary) — **this applies to TOP-LEVEL phases (P0→P1…), NOT the P0.1–P0.7 clusters** (P0 is one phase; firing it per cluster would mean 6 needless interruptions in the most important phase), so P0 instead takes a **cluster-boundary SOFT-stop after ≥5 committed boxes**; the phase-change hard-stop does **NOT** fire if the counter is **0** (all phase boxes were `[!extern]`/`[!]`-skipped); **3 consecutive gate-red pushes = hard-stop+escalate**. (Adapted from the RMA model; digits are the v1 baseline, tunable.)
- **Opt-in box-batching (RMA refinement):** at most **3 sister-boxes** in the same cluster, no sub-boxes / no cross-deps, trivial repetition only (e.g. the structurally-identical per-engine rows in P0.7) ⇒ **1 build-commit + 1 dual-review over the combined diff + 1 check-off**, counter `+= N`; without this rule the loop either wastes dual-review cycles on identical rows or batches implicitly with no rule. **Dual-review fires once per TOP-box** (over the combined sub-box diff), not per sub-box.
- **Spec-contradiction hard-stop:** a **spec-internal contradiction** (two §§ disagree) is an **unconditional hard-stop+escalate regardless of P-severity** — the Build-Loop is downstream of the spec and cannot pick a side. The dual-review prompt instructs **both reviewers to flag a spec §-cross-reference inconsistency as a SPEC-CONTRADICTION finding class above P0** (unconditional hard-stop+escalate, **never a working-tree fix**).
- **Pattern-lookup before escalating a P0:** grep the codebase/process docs for an established pattern first; escalate only if none exists (so a routine choice with a precedent isn't escalated).
- **Per-box status-line format** (the RMA Master-Prompt *output-discipline* section — NOT RMACLAUDE §11, which is the 9-point Definition-of-Done): one line per box — `P0.X done — <summary>, gates green, SHA <short>, Review: P0=N P1=N P2=N P3=N` + a separate `Co-Pilot: N item(s)` block for P1+ findings — so session progress + escalation signals are scannable.
- **8-vs-9 DoD derivation (so the confusion the RMA gap-check flagged cannot recur):** the ConvertIA 8-point DoD is RMACLAUDE-§11's 9 points with the **RLS/tenant, audit-WORM, and migrations rows DROPPED** (no multi-tenant/DB in an offline desktop app) and the **`engines.lock`+SBOM row and the §0.11+§5 threat-class row ADDED** (the two ConvertIA-specific production facts). The canonical 8-point list lives in `build-loop.md`; plan-lint check 14 holds all three copies identical.
- **Convergence-report content (end of session / zero open boxes):** boxes completed + commit SHAs + the **consolidated `[!extern]` list = the owner's action list**, so the owner has a single scannable hand-off.
- **`build-loop.md` is the single canonical home of the 8-point DoD** (the build-gates.md G1 DoD bullet and this P0.6 transcription both cite it); **plan-lint check 14 (DoD-list parity) asserts the G1, P0.6, and canonical `build-loop.md` DoD lists are item-count + item-identifier identical** (the exact "8 vs 9 DoD points" drift the RMA gap-check had to fix — now a machine invariant, not a prose promise), and **check 15 asserts the P0.6 hard-stop / token-Notbremse numbers (~8 soft, ~12 hard, P0 cluster soft ≥5, 3 push-failures) appear verbatim in `build-loop.md`** so the loop cannot silently run on stale thresholds. The §0 push-exit-code / `gh run watch --exit-status` / session-start commands are specified concretely in `build-loop.md`, and the **first-push-to-empty-remote fail-open note is repeated verbatim in the `build-loop.md` session-start section** (so session 1 on a fresh repo does not misread the no-base full-run as a red push and trip the 3-failures escalation).
- **Crash-recovery procedure (build-loop.md, currently absent):** **(a)** a partial staged state → `git reset HEAD` + re-read the box (no half-staged commit); **(b)** committed-but-CI-red → a NEW commit fixing it, **never amend a pushed commit**; **(c)** pushed-but-not-checked-off → the normal open-box scan catches it (the check-off is idempotent on retry); **(d)** push is idempotent on retry. So a session crash mid-box is recoverable without manual surgery.
- **Dual-review divergence-resolution rule (build-loop.md, currently absent):** a **P0/P1 GO-vs-NOGO divergence is treated as NOGO (the stricter reviewer wins)**; a **P2/P3 divergence is resolved by the loop with a recorded `[Build-Session-Entscheidung]` rationale** — UNLESS it is a spec-contradiction, which is the unconditional hard-stop+escalate above (the loop is downstream of the spec and cannot pick a side).
- **Conflict rule** (written as a build-loop rule): **SSOT > spec > security/process docs > code > conversation.**
- Escalation rules — when the Build-Loop escalates to the Co-Pilot session; the hard-stop conditions (concrete box-counter / token-notbremse numbers); the "decide-it-yourself" default + tagging (`[Build-Session-Entscheidung]`).
- **Vuln-response runbook** (`docs/process/vuln-response.md`) — because the app ships known-vulnerable-class C/C++ decoders against untrusted files with **no auto-update**, the only path a security fix reaches a user is a new full release: advisory (G17b/upstream) → Build-Loop escalates to Co-Pilot → bump the `engines.lock` pin → re-run the §6.5 reliability gate → new release; matches the `SECURITY.md` "no SLA, best-effort" posture with an actual triage process. **Severity threshold (stated in `SECURITY.md`):** a CVE with **CVSS ≥ 7 on an engine code path ConvertIA actively exercises for a §04 format** → MUST escalate + block the next release until bumped or triaged not-exercised (G17b release-tier rule), so a CVSS≥9 PoC in an actively-exercised FFmpeg/poppler path can't sit shipped for weeks. **Beyond engine CVEs, the runbook also covers:** (a) **own-code (non-engine) vulns** + the coordinated-disclosure intake→embargo→fix→release loop (via the private-advisory template, §6.8); (b) a **signing-key compromise/loss path** — the human-readable "this key is retired/compromised" commit IS the revocation channel for an offline no-phone-home app (an authored artifact), plus an offline encrypted key+passphrase backup/custody note (the §6.2.3 rotation policy is the silent-swap defence; this is the incident path it doesn't cover).
