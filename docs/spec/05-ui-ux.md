## 5.1 Frontend stack & structure

### Stack `[DECIDED]`
React 19 + TypeScript (strict, **no `any`** per the platform rule) + Tailwind CSS,
built by Vite, running inside the Tauri WebView. Pinned versions are owned by
**┬¦0.8** (not re-pinned here). The frontend is **UI only** ŌĆö it holds no
conversion logic, no filesystem access, and no engine knowledge; every effectful
operation crosses the IPC boundary into the Rust core (┬¦0.3 two-tier model). The
WebView runtime varies per OS (WebView2 / WKWebView / WebKitGTK, ┬¦0.3.1) ŌĆö the
frontend therefore targets the **intersection** of those engines and avoids
bleeding-edge CSS/JS that drifts across them (rendering-drift testing is ┬¦6.4).

### State management choice `[DECIDED ŌĆö Zustand, recommendation]`
The app is a **single linear wizard with one batch in flight at a time** (the
SSOT batch rule allows only one source format per run; ┬¦1.3). That makes the
state small and mostly a finite-state-machine plus a per-item progress list.

- **Screen-flow state** ŌåÆ a hand-rolled **reducer-based finite-state machine**
  (`state/machine.ts`, plain TS, see ┬¦5.2). A typed discriminated-union `State`
  + `dispatch(action)` is preferred over a library (XState would work but is
  weight the SSOT *lightweight* principle doesn't justify for ~9 states).
- **Shared app store** ŌåÆ **Zustand** `[recommendation]`. Rationale: tiny, no
  provider boilerplate, ergonomic selectors (avoids needless re-render of a
  1000-row progress list), and trivially testable in Vitest. The store holds the
  machine state, the collected batch, the chosen target+options, the resolved
  destination preview, and the live progress map. Redux Toolkit is heavier than
  warranted; bare Context+useReducer re-renders too broadly for the large
  progress list (┬¦1.10 virtualisation). **This is the one genuinely
  substitutable decision in 5.1** ŌĆö flagged as a recommendation, not owner-level
  blocking; any equivalent minimal store is acceptable so long as it keeps the
  IPC-fa├¦ade rule and selector-granularity.
- **No data-fetching/cache library** (React Query etc.) ŌĆö there is no server; all
  data crosses IPC and is push-driven via a Channel (┬¦5.8), not request/cache.

### Shared types with Rust
The RustŌåöTS type bridge (so `Batch`, `DetectionOutcome`, `ConversionJob`,
`Target`, `RunResult`, error/lossy payloads, the progress event union) are
**typed end-to-end with no `any`** ŌĆö mechanism owned by **┬¦0.4.5** (DECIDED:
**tauri-specta** + specta; CI drift check in ┬¦06). This section **consumes**
those generated types from `src/lib/ipc/bindings.ts` (┬¦0.7/┬¦0.4.5) and re-decides
nothing.

## 5.2 Screen states / flow ŌĆö **owner of the frontend state machine**

The UI is a finite-state machine. Each state is a discriminated union variant
carrying exactly the data that state needs; transitions are driven by user
actions and by inbound IPC results/events (┬¦5.8). The backend is the source of
truth for *facts* (what was detected, what targets exist, where output will go,
per-item outcome); the machine only sequences the user through them.

## 5.4 Drag-and-drop & input ŌĆö **the native file-drop boundary (a ┬¦0.4 fact)**

### Native file-drop (the load-bearing constraint)
**HTML5 drag-and-drop inside a Tauri WebView does not expose real filesystem
paths** (this is the ┬¦0.4 boundary fact). Intake therefore uses **Tauri's native
drag-drop event**, not the DOM `drop` event:

```ts
// src/lib/ipc/events.ts ŌĆö the ONLY place this is wired (┬¦0.7 path)
import { getCurrentWindow } from '@tauri-apps/api/window';

const unlisten = await getCurrentWindow().onDragDropEvent((e) => {
  switch (e.payload.type) {
    case 'enter':
    case 'over':  setDragActive(true); break;        // visual affordance only
    case 'leave': setDragActive(false); break;
    case 'drop':  setDragActive(false);
                  // e.payload.paths: absolute path strings (files AND folders).
                  // C1 (┬¦0.4.1) REQUIRES a frontend-generated collectingId + origin
                  // (README DECIDED: "the frontend generates CollectingId and passes
                  // it as a C1 argument"), so C13 can cancel this in-flight walk.
                  const collectingId = crypto.randomUUID() as CollectingId;
                  ipc.ingestPaths({
                    paths: e.payload.paths,
                    origin: 'drop',          // IntakeOrigin (REQUIRED, ┬¦0.4.1 C1)
                    collectingId,
                    onScan,                  // ScanProgress Channel ŌĆö ALWAYS passed (non-optional, ┬¦0.4.1);
                                             // created per ┬¦5.8, the frontend subscribes only for a long walk
                  });                          // ŌåÆ ┬¦0.4 C1 ingest_paths
                  break;
  }
});
```

- `event.payload.paths` is an **array of absolute path strings** (not DOM `File`
  objects). Folders arrive as a directory path; **folder recursion runs in Rust**
  (┬¦1.1) ŌĆö the WebView cannot and must not enumerate a directory.
- The window must have `dragDropEnabled: true` (Tauri default) for the native
  event to fire; the DOM-level DnD that *would* hijack it is left disabled.
- `onDragDropEvent` is a method on the **Window** object (`getCurrentWindow()` from
  `@tauri-apps/api/window`) in Tauri v2; verify the exact import/method against the
  pinned Tauri v2 minor at build time (┬¦6.1) ŌĆö the v1 `fileDropEvent` API is gone.
- **Known gotchas to handle (┬¦6.4 test items):** native drag-drop events can
  duplicate / report differing webview ids on some platforms, and path payloads
  have shifted between Tauri patch versions ŌĆö `src/lib/ipc/events.ts` de-duplicates by
  the `paths` set per drop and treats the backend's frozen-set de-dup (┬¦2.4) as
  the authority; the frontend never assumes the path list is unique or canonical.
- The drop only *hands paths to the backend*; the **frozen source set** (┬¦2.4),
  detection (┬¦1.2) and grouping (┬¦1.3) all happen Rust-side. The UI transitions
  `Idle ŌåÆ Collecting` on `drop` and waits for the backend's collected summary.

### Native drop in a NON-`Idle` state `[DECIDED]`
`onDragDropEvent` is window-global, so a `drop` can arrive in **any** UI state, not just
`Idle`. The per-state policy (so a Phase-3 reducer has a definite branch for every state):
- **Pre-run, not-yet-converting states ŌĆö `Confirm` (3), `Targets` (4), `Destination` (5):**
  a drop is treated as a **RE-DROP**: the not-yet-run pre-run state is **discarded** and the
  machine returns to **`Collecting`** with the newly-dropped paths (a fresh C1 freeze). No
  work has started, the user clearly means "actually, convert *these* instead", and there is
  no partial result to lose. (Same shape as the ┬¦5.2 row-9 MixedDropRefusal re-drop.)
- **`Summary` (8) (post-run):** a drop is a **re-drop ŌåÆ `Collecting`** as well ŌĆö it is the
  ergonomic equivalent of "convert more" with files already in hand (the finished run is
  terminal and kept on disk; nothing is lost).
- **`MixedDropRefusal` (9) / `UnsupportedNotice` (10) (pre-flight refusal/notice states):** a
  drop is treated as a **re-drop ŌåÆ `Collecting`** (dismiss-then-drop) ŌĆö these are pre-flight
  states with no run in flight and nothing to lose; a fresh drop clearly means "try these
  instead", so it dismisses the notice and freezes the new paths (a fresh C1 freeze), exactly
  like the MixedDropRefusal re-drop affordance (state 9 renders an **active** re-drop
  `DropZone` for precisely this; state 10's drop is the same branch even though its only
  visible control is Dismiss). This closes the ┬¦5.4 "a definite branch for every state"
  promise ŌĆö **no state is left with an undefined drop behaviour**.
- **`RerunPrompt` (6):** the focus-trapped decision modal **ignores** a native drop while it
  is open (the user must resolve Skip / Make-fresh-copy first). The drop is **silently
  ignored** ŌĆö the drop event is **consumed with no state change and no visual feedback**;
  the alertdialog remains open and the held plan is untouched (no toast, no flash, no
  `BusyNotice`).
- **`Converting` (7) / `Converting (CancellingŌĆ”)` (7a):** a drop is **ignored** ŌĆö the
  `DropZone`'s `disabled`-while-converting guard is inert here and the one-batch-at-a-time
  model (┬¦1.3) forbids a mid-run intake (mirrors the ┬¦7.1 refuse-busy posture for launch
  intake; no `BusyNotice` is needed for a same-window drop, it is simply a no-op).
- **`AppCloseRequested` (11) / `AppFault` (12):** **ignored** ŌĆö a quit-confirm overlay and a
  fault-recovery screen are not intake surfaces.

### File picker (parity path)
Click on the DropZone (or the **Ctrl/Ōīś + O** accelerator, ┬¦5.10) invokes the **intake
picker command `C2a pick_for_intake`** (`kind: files`). The **choose-folder affordance**
is a **secondary "or choose a folder" link on the DropZone** (and the **Ctrl/Ōīś + Shift +
O** accelerator, ┬¦5.10) that calls the **same C2a with `kind: folder`** ŌĆö it surfaces in
`Idle` next to the files picker, so a folder is as discoverable as a file. The native
dialog is opened **Rust-side via
`DialogExt`** inside the C2a handler `[DECIDED]` ŌĆö there is **no JS `open({...})` call
and no `dialog:allow-open` WebView grant** (┬¦0.10). The picked paths are funnelled
**straight into the C1 `ingest_paths` freeze Rust-side**, and C2a returns the **same
`CollectedSet`** the WebView would get from a drop ŌĆö so **no raw FS path ever reaches
the WebView**; the UI transitions `Idle ŌåÆ Collecting` and renders the returned summary
exactly as for a drop (one intake funnel, ┬¦1.1). **Cancelling the dialog is a clean
no-op**: C2a returns `CollectedSet::Empty`, the UI stays in `Idle`, no error, no
`Collecting` transition.

The **Change-destination** picker is the *separate* command **`C2b pick_destination`**
(folder-only, also Rust-opened via `DialogExt`): it **returns the chosen folder
`PathBuf` to the WebView**, which carries it into **C5 `set_destination`**. This is the
one picker whose path *does* transit the WebView ŌĆö unavoidable, since the destination is
a WebView-held choice ŌĆö and is acceptable (a *write* destination, bounded by ┬¦2.1, see
┬¦0.11 T2a). Capability scope is owned by **┬¦0.10**; this section only invokes within it.

### Keyboard parity
Every result reachable by drop/pick is reachable by keyboard alone (SSOT DoD
gate). The DropZone is a focusable, `role="button"` element; **Enter/Space**
activates the picker; the full accelerator map is **┬¦5.10**. There is no
keyboard-only dead end in the flow.

### Launch-time intake
Paths can also arrive via OS launch entry points (Open-with / argv / macOS
open-doc) ŌĆö posture owned by **┬¦7.8**; they feed the **same** `ingest_paths` (C1)
funnel, so the UI handles a launch-with-files identically to a drop (machine
enters `Collecting` at startup instead of `Idle`).

## 5.5 Design system

### Intent
"**Modern > plain**" (SSOT *Design Intent*): uncluttered, contemporary, a little

(SSOT ┬¦9 *Not a gate*); the *structure* and tokens here are the contract, the

### Tokens `[DECIDED ŌĆö token contract; values are placeholders]`

Tailwind via the theme config so components use semantic Tailwind classes, not
raw hex. Token groups:

- **Colour (semantic, not literal):** `--bg`, `--surface`, `--surface-raised`,
  `--border`, `--text`, `--text-muted`, `--accent` (the one brand accent),
  `--accent-contrast`, plus state colours `--success`, `--warn`, `--danger`,
  `--info`. Lossy/divert notes use `--info`/`--text-muted` (calm, **not**

  keyboard-equivalent (the picker).
- **Logical focus order** following the visual wizard order; focus is **moved to
  the new primary element on each state transition**. Two named rules govern the

## 5.8 IPC integration & frontend async model

The **command/event contract is owned entirely by ┬¦0.4** (every command name,
request/response payload, error shape, cancellation token, and every event/
Channel and its payload). This section defines only the **frontend async
behaviour** that *consumes* that contract. Nothing here re-declares a command
name or payload shape ŌĆö `src/lib/ipc/commands.ts` and `src/lib/ipc/events.ts` are the
typed wrappers; feature code calls those.

### Command/response model
- All effectful operations are **`invoke()` calls** into the Rust core, awaited as
  Promises, typed via ┬¦0.4.5 generated types (no `any`). The full set of commands the
  frontend calls (names defined in ┬¦0.4, not invented here): `ingest_paths` (C1),
  `pick_for_intake` (C2a ŌĆö DropZone "browse" / keyboard picker ŌåÆ returns `CollectedSet`)
  and `pick_destination` (C2b ŌĆö Change-destination directory picker ŌåÆ returns `PathBuf` for C5), ┬¦5.4,
  `get_targets` (C3), `plan_output` (C4), `set_destination` (C5), `start_conversion`
  (C6), `cancel_run` (C7), `get_run_summary` (C8 ŌĆö idempotent summary re-fetch by a
  **fresh listener AFTER the run terminated**; **reload-during-run is NOT a supported
  recovery path on macOS in v1** ŌĆö a known open Tauri crash, ┬¦0.4.4 ŌĆö so a mid-stream
  reload surfaces as `AppFault`, not a silent recovery), `open_path` (C9 ŌĆö open-folder/open-file, ┬¦7.7), `open_project_page`
  (C10 ŌĆö the About "open Releases" link, ┬¦5.9/┬¦7.6), `get_app_info` (C11 ŌĆö About data,
  ┬¦5.9), `get_engine_health` (C12 ŌĆö drives disabled/omitted target tiles, ┬¦5.2/┬¦3.4),
  `cancel_ingest` (C13 ŌĆö the Collecting cancel-collect control, ┬¦5.2/┬¦5.10). The
  frontend treats these as opaque typed RPCs.
- Long-running work (the conversion run) must **not** block on a single Promise
  resolving at the end (Cloudflare-100s-style hangs don't apply locally, but a
  60-minute batch resolving one Promise at the end gives no progress) ŌĆö instead
  the **`start_conversion` command returns quickly** and progress flows over a
  **Channel** (below). The pattern mirrors the platform's "respond immediately,
  stream/poll the rest" posture.

#### IPC call-timing on the Confirm ŌåÆ Targets ŌåÆ Destination flow `[DECIDED]`
The ┬¦5.2 state machine leaves *which transition* fires C3/C4/C5 implicit; pinned here so
the Phase-3 task list is unambiguous:
- **C3 `get_targets`** is called **on the Confirm ŌåÆ Targets transition (3 ŌåÆ 4)** ŌĆö the UI
  enters `Targets` only **after** C3 resolves (so the FormatPicker always has its tiles +
  pre-highlighted default before it is shown). C3 is a pure function of detection (no
  engine), so it is cheap.
- **C4 `plan_output`** is called **once a target is selected** ŌĆö i.e. **eagerly on entry
  to `Targets` with the pre-highlighted default already selected**, then **re-called
  (debounced ~150 ms) on any target change or option change** so the `DestinationBar`
  "will save to ŌĆ”" line, the divert preview, the `rerun` prompt, and the
  `preflight.up_front_fail` verdict never go stale. The SSOT requires a destination preview
  **before** Convert, so C4 must have resolved before the Convert button is enabled.
  **After a C5 destination change, a subsequent target/option change still re-runs C4
  (┬¦0.4.1) ŌĆö but that C4 call carries the C5-resolved destination in its `destination`
  argument (the held session destination, not the persisted default), so C4 never resets
  the destination away from the user's C5 choice.** Destination authority stays with C5.
- **C5 `set_destination`** is called **only when the user changes the destination** via
  the C2b `pick_destination` picker (Change-destination); it re-evaluates the
  destination-dependent preflight and **carries `rerun` through unchanged** (┬¦2.5.1
  destination-independence). It is **not** part of the default forward path.

#### Persisted `lastDestinationMode` ŌåÆ C4's default destination `[DECIDED]`
┬¦7.4 persists a `lastDestinationMode` prefs key (`"beside-source"` | `"<absolute path>"`,
default `beside-source`). Who reads it, when, and how it reaches C4 is pinned here so a
returning user's `DestinationBar` initial state is derivable:
- **WHO + WHEN:** the **frontend reads `lastDestinationMode` from `tauri-plugin-store`**
  (the ┬¦7.4.2 `store:default` mechanism, the same door it uses for `theme`/`verboseLog`)
  **at startup / app-store hydration** ŌĆö JS-side via the store plugin, not a new Rust
  command (no IPC surface is added; the store read is the existing capability). It is held
  in the ┬¦5.1 store as the session's default destination.
- **HOW it reaches C4:** the persisted value is the **`destination` argument on C4
  `plan_output`'s FIRST Targets-entry call** (the eager call on the `3ŌåÆ4` transition above)
  ŌĆö `"beside-source"` ŌåÆ `DestinationChoice::BesideSource`; `"<absolute path>"` ŌåÆ
  `DestinationChoice::ChosenRoot(path)`. So a returning user who last chose a custom folder
  sees the `DestinationBar` pre-populated with it (the "will save to ŌĆ”" line reflects it)
  without re-picking.
- **A stored absolute path is a re-validated HINT, never a guarantee (┬¦7.4.1):** C4's
  per-volume preflight (┬¦1.10) and the ┬¦2.7 writability/divert check re-validate the stored
  path **at use time**; if it has become read-only/gone, the plan **falls back to

### Cancellation ŌĆö optimistic vs confirmed round-trip
SSOT *Visible progress, cancellable* + the cancel **mechanism owned by ┬¦1.7**
(process-group kill, no-partial-leftover) means cancel is **not** instant and
must be honest:

1. User hits Cancel (button or **Esc**, ┬¦5.10) in `Converting`.
2. UI **optimistically** flips to a "CancellingŌĆ”" affordance and **disables**
   Cancel (no double-cancel) ŌĆö **a second Esc (or Cancel click) while in the
   "CancellingŌĆ”" state is silently ignored** (no second `cancel_run`, no UI change),
   so the user cannot double-cancel; the UI does **not** fabricate a finished state.
3. Frontend calls the **cancel command** (name ┬¦0.4) with the `runId` /
   cancellation token (┬¦0.4 token shape).
4. The backend stops the queue, kills the in-flight engine (┬¦1.7), cleans the
   in-progress temp (┬¦2.6), and emits terminal `ItemFinished{outcome:Cancelled}`
   for the stopped item + `RunFinished` over the **same Channel** (┬¦0.4.2 names).
5. Only on that **confirmed** `RunFinished` does the UI move to `Summary`
   (partial): items already finished are **kept** (SSOT), the in-progress one is
   shown `Cancelled`, no partial leftover. The UI never claims "cancelled" before
   the backend confirms the kill+cleanup landed.

### `RunStarted.willReencode` consumption
`RunStarted` (┬¦0.4.2) carries `willReencode: boolean` (the ┬¦2.9.2 best-effort

kind (┬¦2.9.2), which the FormatPicker renders as a passive note the moment that target
is selected. `willReencode` is therefore the **during-`Converting` confirmation**, not
the note's first or only source. **The carry-over store field (state 4 ŌåÆ 7) `[DECIDED]`:**
the note text is held in a single store field **`pendingVideoReencodeNote: string | null`**
ŌĆö **set** (to the ┬¦2.9 `video_reencode` string) when C3 returns `Target.lossy =
video_reencode` for the selected target in state 4, **carried unchanged through states
4 ŌåÆ 5 ŌåÆ 6 ŌåÆ 7** (it survives the FormatPicker ŌåÆ DestinationBar ŌåÆ optional RerunPrompt
transitions so it does not have to be recomputed), and **read by `ConvertingNote`** in
state 7. On `RunStarted` the store reconciles it with `RunStarted.willReencode`: if `true`
it **keeps** `pendingVideoReencodeNote` (and `ConvertingNote` renders it ŌĆö re-surfacing it

it **sets `pendingVideoReencodeNote = null`**, which **silently clears** the pre-shown
worst-case note. It is the **same ┬¦2.9 string** throughout (state-4 note and state-7 banner
share one value, never two divergent strings). So the note's lifecycle is: **set at state 4
(C3 `Target.lossy`) into `pendingVideoReencodeNote` ŌåÆ carried 4ŌåÆ7 ŌåÆ confirmed (kept) or
cleared (set `null`) by `RunStarted.willReencode`**; the authoritative per-item lossy
outcome still comes from the ┬¦1.12 summary.

**Back-navigation reset (4ŌåÆ3) `[DECIDED]`.** ┬¦5.2/┬¦5.10 allow a `Targets (4) ŌåÆ Confirm
(3)` back-navigation (the Back button / Ctrl/Ōīś+Backspace). On that **4ŌåÆ3** transition the
reducer **resets `pendingVideoReencodeNote = null`** ŌĆö otherwise a re-encode note set for a
*prior* target could persist and mislead if the user re-enters Targets and picks a
**different, non-lossy** target. On the next **3ŌåÆ4** the field is **re-derived from the
fresh C3 response** (set only if the newly-selected target's `Target.lossy = video_reencode`).
So the field always reflects the *current* selection, never a stale prior one; the same
