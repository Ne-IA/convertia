## 5.1 Frontend stack & structure

### Stack `[DECIDED]`
React 19 + TypeScript (strict, **no `any`** per the platform rule) + Tailwind CSS,
built by Vite, running inside the Tauri WebView. Pinned versions are owned by
**¬ß0.8** (not re-pinned here). The frontend is **UI only** ‚Äî it holds no
conversion logic, no filesystem access, and no engine knowledge; every effectful
operation crosses the IPC boundary into the Rust core (¬ß0.3 two-tier model). The
WebView runtime varies per OS (WebView2 / WKWebView / WebKitGTK, ¬ß0.3.1) ‚Äî the
frontend therefore targets the **intersection** of those engines and avoids
bleeding-edge CSS/JS that drifts across them (rendering-drift testing is ¬ß6.4).

### Component / folder structure `[DECIDED]`
Frontend code lives under the physical tree owned by **¬ß0.7** (`src/`); the design
**tokens/assets** home is the repo-root **`design/`** sibling (¬ß5.5, *not* under
`src/` ‚Äî the ¬ß0.7 tree is completed to home it in P1.64, and it is the single colour
home G9 invariant (a) exempts). Logical grouping:

**Hard rule:** only `src/lib/ipc/**` (the ¬ß0.7 path) imports the Tauri IPC surface ‚Äî
`@tauri-apps/api` **or** any `@tauri-apps/plugin-*` JS package (a plugin package wraps a

`@tauri-apps/plugin-log` bridge homed in `src/lib/ipc/log.ts`). Components and state talk
to a typed fa√ßade, so the IPC contract (¬ß0.4) has exactly one consumer and the "no direct
`fetch`/raw-`invoke` in feature code" discipline (platform auth-pattern analogue) is
enforceable by lint (G5).

### State management choice `[DECIDED ‚Äî Zustand, recommendation]`
The app is a **single linear wizard with one batch in flight at a time** (the
SSOT batch rule allows only one source format per run; ¬ß1.3). That makes the
state small and mostly a finite-state-machine plus a per-item progress list.

- **Screen-flow state** ‚Üí a hand-rolled **reducer-based finite-state machine**
  (`state/machine.ts`, plain TS, see ¬ß5.2). A typed discriminated-union `State`
  + `dispatch(action)` is preferred over a library (XState would work but is
  weight the SSOT *lightweight* principle doesn't justify for ~9 states).
- **Shared app store** ‚Üí **Zustand** `[recommendation]`. Rationale: tiny, no
  provider boilerplate, ergonomic selectors (avoids needless re-render of a
  1000-row progress list), and trivially testable in Vitest. The store holds the
  machine state, the collected batch, the chosen target+options, the resolved
  destination preview, and the live progress map. Redux Toolkit is heavier than
  warranted; bare Context+useReducer re-renders too broadly for the large
  progress list (¬ß1.10 virtualisation). **This is the one genuinely
  substitutable decision in 5.1** ‚Äî flagged as a recommendation, not owner-level
  blocking; any equivalent minimal store is acceptable so long as it keeps the
  IPC-fa√ßade rule and selector-granularity.
- **No data-fetching/cache library** (React Query etc.) ‚Äî there is no server; all
  data crosses IPC and is push-driven via a Channel (¬ß5.8), not request/cache.

### Shared types with Rust
The Rust‚ÜîTS type bridge (so `Batch`, `DetectionOutcome`, `ConversionJob`,
`Target`, `RunResult`, error/lossy payloads, the progress event union) are
**typed end-to-end with no `any`** ‚Äî mechanism owned by **¬ß0.4.5** (DECIDED:
**tauri-specta** + specta; CI drift check in ¬ß06). This section **consumes**
those generated types from `src/lib/ipc/bindings.ts` (¬ß0.7/¬ß0.4.5) and re-decides
nothing.

## 5.2 Screen states / flow ‚Äî **owner of the frontend state machine**

The UI is a finite-state machine. Each state is a discriminated union variant
carrying exactly the data that state needs; transitions are driven by user
actions and by inbound IPC results/events (¬ß5.8). The backend is the source of
truth for *facts* (what was detected, what targets exist, where output will go,
per-item outcome); the machine only sequences the user through them.

> `MixedDropRefusal` (state 9) and `Unsupported` (state 10) are **pre-flight** ‚Äî
> nothing is converted. A single bad/unreadable/too-big item discovered **during**
> `Converting` is **not** a state change: it surfaces as that item's terminal
> `Failed`/`Skipped` row in `ProgressList` and carries into `Summary`; the batch
> keeps going (¬ß1.9/¬ß2.8). Two visually similar "something's wrong" outcomes, two
> different mechanisms ‚Äî the UI must not conflate them.

## 5.4 Drag-and-drop & input ‚Äî **the native file-drop boundary (a ¬ß0.4 fact)**

### Native file-drop (the load-bearing constraint)
**HTML5 drag-and-drop inside a Tauri WebView does not expose real filesystem
paths** (this is the ¬ß0.4 boundary fact). Intake therefore uses **Tauri's native
drag-drop event, handled CORE-side** ‚Äî not the DOM `drop` event ‚Äî and per the
**2026-07-06 owner ruling (core-owned paths)** the dropped paths never enter the
WebView at all:

- **The drop is a Rust window event:** the core's **`WindowEvent::DragDrop`** handler
  receives the dropped paths (files AND folders, real `PathBuf`s) and feeds them into
  the **¬ß7.8.1 intake funnel** ‚Äî the same funnel every other intake source uses
  (launch-arg, second-instance, Open-with, the C2a picker) ‚Äî which enforces the
  uniform **¬ß7.1.1 refuse-busy gate**, stashes the set into the core-side
  **`PendingIntake`** buffer, and emits the **payload-less `app://intake` nudge**
  (¬ß0.4.2). The ¬ß5.8 consumption then calls **C1 `drain_intake`** (a fresh
  frontend-generated `collectingId` + the `onScan` Channel ‚Äî so C13 can cancel the
  in-flight walk) and drives ¬ß5.2 from the returned `CollectedSet`. Folder recursion
  runs in Rust (¬ß1.1) ‚Äî the WebView cannot and must not enumerate a directory.
- **The WebView renders the drag-over affordance from DOM drag events ONLY:** the
  DropZone's `dragActive` lift/glow (¬ß5.5) is driven by DOM
  `dragenter`/`dragover`/`dragleave` styling hooks ‚Äî pure visual state, **never** a
  path source (a Tauri WebView's DOM drop payload carries no real path anyway, and
  per the ruling it would be forbidden intake even if it did). The window keeps
  `dragDropEnabled: true` (Tauri default) so the native event fires core-side.
- **Known gotchas to handle (¬ß6.4 test items), now CORE obligations:** native
  drag-drop events can duplicate / report differing webview ids on some platforms,
  and path payloads have shifted between Tauri patch versions ‚Äî the core's
  `DragDrop` handler de-duplicates by the `paths` set per drop and treats the
  frozen-set de-dup (¬ß2.4) as the authority; nothing assumes the path list is
  unique or canonical. (These duties sat in the frontend `events.ts` wiring in the
  pre-revision design; they move core-side with the handler.)
- The drop only *hands paths to the core funnel*; the **frozen source set** (¬ß2.4),
  detection (¬ß1.2) and grouping (¬ß1.3) all happen Rust-side. The UI transitions
  `Idle ‚Üí Collecting` when its nudge-triggered C1 drain is issued and waits for the
  backend's collected summary.

> **Supersede-note `[DECIDED ‚Äî the 2026-07-06 owner ruling (core-owned paths)]`.**
> The pre-revision ¬ß5.4 wired the drop in the WebView: an `events.ts`
> `onDragDropEvent` subscription read `e.payload.paths` ("absolute path strings")
> and called C1 (`ingest_paths { paths, origin, collectingId, onScan }`). That
> wiring is REMOVED from the design: **no FS path crosses the IPC wire or an
> `app://` event, in either direction** ‚Äî JSON is UTF-8-only, so a serde wire
> carrying paths fails on non-UTF-8 names (breaking ¬ß2.10.1), and a WebView that
> never receives a path is a WebView that can never *supply* one (the
> WebView-named-path threat class dissolves). The wire carries IDs + lossy display
> strings; real paths live in core registries. The P3 wire-revision boxes own the
> code change.

### Native drop in a NON-`Idle` state `[DECIDED]`
The native drop is window-global, so a drop can arrive in **any** UI state, not just
`Idle`. The per-state policy is enforced at **two layers** (a Phase-3 reducer still has
a definite branch for every state ‚Äî the outcomes below restate the pre-revision
per-state decisions in the funnel model's homes):
- **Core-side (the primary gate, ¬ß7.8.1):** the funnel applies the **¬ß7.1.1 refuse-busy
  gate** to a drop exactly as to a second-launch hand-off ‚Äî **mid-conversion
  (`Converting` (7)/(7a)) the dropped paths are dropped core-side**: no stash, no
  nudge, nothing reaches the UI (the one-batch-at-a-time model ¬ß1.3 and the ¬ß2.4
  freeze stay unambiguous; the drop does not merely *mirror* the ¬ß7.1 refuse-busy
  posture ‚Äî it now **is** that posture, one gate for all intake). When not busy, the
  funnel stashes + nudges.
- **UI-side (from its OWN state ‚Äî the ¬ß5.8 consumption rule):** on the nudge, the
  frontend consults its machine state:
  - **Fresh-intake states ‚Äî `Idle` (1), `Confirm`/`Targets`/`Destination` (3/4/5),
    `Summary` (8), `MixedDropRefusal` (9), `Unsupported` (10):** it **drains** (C1)
    and enters `Collecting`, discarding any not-yet-run pre-run state ‚Äî the same
    "actually, convert *these* instead" reading as before (no work has started,
    there is no partial result to lose); a `Summary` drop is the ergonomic "convert
    more" with files already in hand (the finished run is terminal and kept on
    disk); a state-9/10 drop dismisses the notice into a fresh freeze (state 9's
    active re-drop `DropZone` and state 10's dismiss-then-drop equivalence are
    unchanged, ¬ß5.2/¬ß5.3).
  - **Non-intake states ‚Äî `Collecting` (2), `RerunPrompt` (6), `Converting`
    (7)/(7a) (core-gated, so only a *leaked* nudge can arrive there),
    `AppCloseRequested` (11), `AppFault` (12):** it does **not** drain; it renders
    the passive **`BusyNotice`** Banner from its own state (¬ß5.3/¬ß5.8), and the
    stashed set stays pending core-side (consumed by a subsequent drain; a further
    intake in the meantime APPENDS to it, first origin kept ‚Äî the ¬ß7.8.1 no-loss
    accumulation). The payload-less nudge **cannot set-swap** a held collected set or
    plan ‚Äî it carries nothing.

> **Supersede-note `[the 2026-07-06 owner ruling (core-owned paths)]`.** The
> pre-revision per-state drop branches survive with two deliberate deltas: (1) the
> `Converting` no-op moves **core-side** (the funnel's refuse-busy gate ‚Äî the UI
> never sees a mid-run drop); (2) the "silently ignored ‚Äî no toast, no flash, no
> `BusyNotice`" branches for states 6/11/12 are superseded by the calm `BusyNotice`
> Banner ‚Äî the dropped set is now really buffered core-side, so a wholly invisible
> drop would be dishonest; the Banner is passive and non-modal, the alertdialog
> stays open, and the held plan is untouched.

### File picker (parity path)
Click on the DropZone (or the **Ctrl/‚åò + O** accelerator, ¬ß5.10) invokes the **intake
picker command `C2a pick_for_intake`** (`kind: files`). The **choose-folder affordance**
is a **secondary "or choose a folder" link on the DropZone** (and the **Ctrl/‚åò + Shift +

`Idle` next to the files picker, so a folder is as discoverable as a file. The native
dialog is opened **Rust-side via
`DialogExt`** inside the C2a handler `[DECIDED]` ‚Äî there is **no JS `open({...})` call
and no `dialog:allow-open` WebView grant** (¬ß0.10). The picked paths are funnelled

nudge ‚Äî C2a itself walks nothing and returns no set; **C1 `drain_intake` is the single
collection door and the single `onScan` carrier**, ¬ß0.4.1), so **no raw FS path ever
reaches the WebView** ‚Äî under the 2026-07-06 owner ruling (core-owned paths) this is
the universal rule for *every* intake surface, not a C2a-special; the UI transitions

### Keyboard parity
Every result reachable by drop/pick is reachable by keyboard alone (SSOT DoD

## 5.5 Design system

### Intent
"**Modern > plain**" (SSOT *Design Intent*): uncluttered, contemporary, a little
eye candy ‚Äî never busy. Visual polish is **iterative and never release-blocking**
(SSOT ¬ß9 *Not a gate*); the *structure* and tokens here are the contract, the
exact palette is a placeholder the owner finalises (SSOT *Design Intent*: logo,
colours, branding are placeholders for now).

### Tokens `[DECIDED ‚Äî token contract; values are placeholders]`
Design tokens are CSS custom properties in `design/tokens.css`, surfaced to
Tailwind via the theme config so components use semantic Tailwind classes, not
raw hex. Token groups:

- **Colour (semantic, not literal):** `--bg`, `--surface`, `--surface-raised`,
  `--border`, `--text`, `--text-muted`, `--accent` (the one brand accent),
  `--accent-contrast`, plus state colours `--success`, `--warn`, `--danger`,
  `--info`. Lossy/divert notes use `--info`/`--text-muted` (calm, **not**
  `--danger` ‚Äî predictable loss is not an error). Failures use `--danger`.
- **Spacing scale:** 4-px base (`--space-1`=4 ‚Ä¶ `--space-8`=32) ‚Äî Tailwind's
  default scale, kept.
- **Radius:** `--radius-sm/md/lg` (cards, tiles, the DropZone ‚Äî generous rounding
  for the modern feel).
- **Typography:** a single clean UI sans (system stack +
  bundled-offline fallback so it renders identically with **zero network**, per
  the offline invariant ¬ß2.11); sizes `--text-xs ‚Ä¶ --text-2xl`; line-heights;
  weight `regular/medium/semibold`. **Readable contrast & text sizes are a DoD
  accessibility gate** (¬ß5.6), so the *minimum* body size and contrast ratios are
  fixed, not placeholder. **Concrete body floor `[DECIDED]`: `--text-base = 1rem
  (16px)` is the minimum size for body copy** ‚Äî the everyday readable default; the
  larger tokens (`--text-lg`/`--text-xl`/`--text-2xl`) are for headings. **`--text-xs`
  (‚âà0.75rem/12px) and `--text-sm` are for *supplementary labels* (meta, captions,
  fine-print) only ‚Äî never for body copy.** This fixes the SSOT "readable text sizes"
  half of the gate at a defensible AA floor (16px), so the ¬ß6.6 walkthrough has a
  concrete minimum to verify against (the ¬ß6.4.6a axe leg checks contrast, not size).
- **Elevation/shadow, motion** tokens (below).

### Light / dark `[DECIDED ‚Äî both; follow OS by default; theme persists]`
Support **light and dark**, defaulting to the **OS preference**
(`prefers-color-scheme`), resolved into the colour tokens at the root. **Theme
persistence is `[DECIDED]`:** ¬ß7.4 ships the **3-key prefs blob** (`theme` +
`lastDestinationMode` + `verboseLog`) via `tauri-plugin-store`, so a chosen theme **does** persist
across launches (a `theme` of `system` keeps following the OS). Because the `theme`
key persists, **a minimal in-app Light/Dark/System toggle IS provided** (it writes the
`theme` key via the store, ¬ß7.4.2) ‚Äî without one, a persisted non-`system` value could
never be set or reset. (The earlier "no in-app toggle required" line is superseded by
the persistence decision.) The default value is `system` (follow the OS) so the
zero-interaction path is unchanged.

### Motion / eye-candy budget `[DECIDED]`
"A bit of eye candy is welcome" (SSOT) but **restrained and accessible**:
- Subtle transitions on the DropZone (lift/glow on `dragActive`), tile selection,
  drawer open/close, and state transitions (‚â§200 ms, ease-out).
- The progress bar animates smoothly between real values ‚Äî **never** a fake/
  indeterminate crawl on the convert step (SSOT *Visible progress* demands a real
  bar so a long single conversion reads as *working, not hung*). An indeterminate
  spinner is permitted **only** for the brief `Collecting` step.
- **`prefers-reduced-motion` is honoured** (¬ß5.6): all non-essential animation is
  disabled/reduced; progress still updates its value, just without easing.

### Loading / progress chrome
- Determinate `ProgressBar` (per-item) + aggregate batch bar in `Converting`.
- `Collecting` may show an indeterminate indicator + a cancel-collect control.
- No modal blocking overlay during conversion ‚Äî the window stays responsive
  (SSOT *stays responsive regardless of batch or file size*), Cancel always live.

### App chrome layout `[DECIDED]`
A single **persistent slim app-header** (`AppHeader`, ¬ß5.3) frames every ¬ß5.2 state and
is the home for the three otherwise-homeless surfaces: **left** = the `<BrandLogo>`

### Ne-IA logo placeholder `[DECIDED ‚Äî placeholder]`

## 5.6 Accessibility ‚Äî **a v1 ship gate, not polish**

SSOT *For anyone ‚Äî accessible* and the ¬ß9 DoD gate ("basic accessibility works:
keyboard path + readable contrast/sizes") make this **release-blocking**, on par
with no-harm. Concrete requirements:

- **Full keyboard operability.** Every action ‚Äî open picker, confirm batch, pick
  target, open Advanced, change destination, convert, cancel, open folder/file,
  dismiss a refusal, answer the re-run prompt ‚Äî is reachable and operable by
  keyboard alone (map ¬ß5.10). No mouse-only affordance exists. The drop area has a
  keyboard-equivalent (the picker).
- **Logical focus order** following the visual wizard order; focus is **moved to
  the new primary element on each state transition**. Two named rules govern the
  Targets/Destination (state 4/5) view `[DECIDED]`:
  - **On entering Targets (3‚Üí4):** focus moves to the **pre-highlighted
    (default-selected) tile in the FormatPicker radiogroup** ‚Äî the `aria-checked="true"`
    tile that is the active radio, so a keyboard user lands on the live default.
  - **When the DestinationBar becomes visible within state 4/5** (the C4 `plan_output`
    response resolves and the "will save to ‚Ä¶" line + Convert button render, ¬ß5.2 row
    4/5): focus moves to the **Convert button**, so a keyboard user can proceed without
    hunting once the destination is shown. (It does not steal focus on every debounced
    C4 re-call ‚Äî only the **first** time the DestinationBar/Convert button appears for
    this batch; subsequent target/option changes leave focus where the user put it.)
- **Converting (7) and Collecting (2) focus-on-entry rules `[DECIDED]`** (the two states
  ¬ß5.6 previously left unstated):
  - **Converting (7):** on entry, focus moves to the **Cancel button** (the one actionable
    control during a run) so a keyboard user can cancel without hunting. When the run ends
    and the machine advances to `Summary`, the ¬ß5.6 Summary focus-on-entry rule takes over.
  - **Collecting (2):** focus moves to the visible **cancel-collect control** (the ¬ß5.2
    Collecting cancel that backs C13, ¬ß5.10) if rendered; if no cancel control is shown yet
    (the walk is too brief to surface one), **prior focus is preserved IF that element still
    exists in the Collecting DOM**. **Orphaned-prior-focus fallback `[DECIDED]`:** if no
    cancel control is rendered **and** prior focus was on an element absent from the
    Collecting state (e.g. it was on the `DropZone`, now hidden/replaced by the scanning
    indicator, or a mouse-drop left no meaningful focus), move focus to the **`Scanning‚Ä¶ N
    files` status region (`role="status"`)** so SR users land on a live, announced element
    rather than a detached/`body`-defaulted focus. **After C13 returns to `Idle`**, focus
    lands on the **DropZone** (the primary Idle affordance), not left orphaned on the
    now-unmounted cancel control.
- **Summary (state 8) focus-on-entry rule `[DECIDED]`:** when the run ends and `Summary`
  mounts, focus moves to ‚Äî in priority order ‚Äî **(1) the first `Failed` row** (announced
  `aria-live="assertive"`) if any item failed; **(2) the `OpenActions` primary button**
  ("Open folder") if **all** items succeeded; **(3) the fully-failed banner's primary
  affordance** (Start over / Convert more) if **every** item failed. For a virtualised
  results list, the target row must be **scrolled into view and rendered before** focus is
  set (never focus a not-yet-mounted virtual row). And
  **trapped inside modals** with **Esc** to close (¬ß5.10). **Focus-restore-on-close is
  scoped to the modals that HAVE a UI trigger `[DECIDED]`:** **RerunPrompt** (triggered by
  the Convert button in state 4/5) and **AboutDialog** (triggered by the About control in
  `AppHeader`) restore focus to **their trigger** on close. **QuitConfirm has NO UI
  trigger** ‚Äî it is raised by `app://close-requested` (an OS close request, ¬ß7.3.2), not a
  UI element ‚Äî so on **Stay/Esc** it returns focus to the **active element in the
  underlying `Converting` state** (the Cancel button, or the focused progress-list row),
  **never** "the trigger" (matching the already-correct ¬ß5.10 QuitConfirm row; restoring
  to a non-existent trigger would be a WCAG 2.4.3 Focus Order failure). The focus-trapped
  modals are exactly three (and only these): the **decision dialogs `role="alertdialog"`**
  ‚Äî **RerunPrompt** + **QuitConfirm** ‚Äî and the **informational dialog `role="dialog"` +
  `aria-modal="true"`** ‚Äî **AboutDialog** (informational, not an alert ‚Äî ¬ß5.6
  decision-states block). **Accessible name (WCAG 4.1.2 / ARIA 1.2) `[DECIDED]`:** every
  one of the three carries an accessible name via **`aria-labelledby`** pointing at its own
  heading element ‚Äî **RerunPrompt** ‚Üí its heading **"Already converted with these
  settings"**; **QuitConfirm** ‚Üí its heading **"Conversion in progress"**; **AboutDialog**
  ‚Üí its **"About ConvertIA"** heading. An `alertdialog` with no accessible name is an
  axe-core a11y-gate (¬ß6.4.6a) failure, so the two `role="alertdialog"` elements MUST name
  their heading (the headings are the ¬ß5.3 component contract). **`MixedDropRefusal` (state
  9) and `UnsupportedNotice` (state 10) are NOT modals `[DECIDED]`** ‚Äî both are **full-screen
  states** (entered by a drop, so no trigger to restore focus to). MixedDropRefusal has its
  own **active re-drop `DropZone`** (the SSOT "re-drop a single format" reading); on entry
  its **heading is announced assertively** (`aria-live="assertive"`) and focus moves to the
  **re-drop `DropZone`**, with a secondary **Dismiss ‚Üí `Idle`**. `UnsupportedNotice` is
  likewise a full-screen pre-flight notice. **Focus-on-entry `[DECIDED]`: focus lands on the
  `Dismiss` button** (its heading is announced via `aria-live="assertive"` but is **not
  focused** ‚Äî it carries `tabindex="-1"` only as a programmatic target, so focusing the

  (Enter/Space on Dismiss, or Esc), focus is **restored to the `DropZone`** (drop again), not

  - **(g) ¬ß5.2 row-6 branch labelling:** the diagram/table branch into RerunPrompt is the

    `aria-valuenow` is set to **100** (Succeeded/Cancelled) or its **last-known** value

      `DropZone` ‚Äî all of which are full-screen-state behaviours, not modal ones. So
      UnsupportedNotice carries **no `role="alertdialog"`**; on entry its heading is
      announced assertively (`aria-live="assertive"`) **while focus lands on the Dismiss
      button** (the heading is `tabindex="-1"`, announced not focused ‚Äî focusing it would make
      Enter a no-op, ¬ß5.6.1 row 10), and on dismiss focus moves to the
      `DropZone` (drop again). **AppFault** (state 12) is likewise a full-screen "something
      went wrong" state with an assertive heading and a single Start-over action.
    Lossy/divert notes announce **politely** (`aria-live="polite"`) ‚Äî they are calm,
    not alarms.
- **Semantics:** the target tiles are a **`role="radiogroup"`** container with
  **`aria-labelledby`** pointing at its heading; each tile is a **`role="radio"`** with
  `aria-checked` (exactly one default checked, ¬ß1.5); the DropZone is `role="button"`;

  than the earlier `tabindex="0"`, which would have broken the single-tab-stop radiogroup
  rule). They are **never** just visual dimming and **never** removed from arrow navigation.

## 5.8 IPC integration & frontend async model

The **command/event contract is owned entirely by ¬ß0.4** (every command name,
request/response payload, error shape, cancellation token, and every event/
Channel and its payload). This section defines only the **frontend async
behaviour** that *consumes* that contract. Nothing here re-declares a command
name or payload shape ‚Äî `src/lib/ipc/commands.ts` and `src/lib/ipc/events.ts` are the
typed wrappers; feature code calls those.

### Command/response model
- All effectful operations are **`invoke()` calls** into the Rust core, awaited as
  Promises, typed via ¬ß0.4.5 generated types (no `any`). The full set of commands the
  frontend calls (names defined in ¬ß0.4, not invented here): `drain_intake` (C1 ‚Äî
  consumes the core `PendingIntake` buffer into the ¬ß1.1 freeze and returns the
  `CollectedSet`; issued on the `app://intake` nudge + once per root-shell mount,
  ¬ß5.4/¬ß7.8.1), `pick_for_intake` (C2a ‚Äî DropZone "browse" / keyboard picker; Rust-side
  dialog feeding the same ¬ß7.8.1 funnel)
  and `pick_destination` (C2b ‚Äî Change-destination directory picker ‚Üí returns `DestinationPicked { destination: DestinationId, display }` for C5), ¬ß5.4,
  `get_targets` (C3), `plan_output` (C4), `set_destination` (C5), `start_conversion`
  (C6), `cancel_run` (C7), `get_run_summary` (C8 ‚Äî idempotent summary re-fetch by a
  **fresh listener AFTER the run terminated**; **reload-during-run is NOT a supported
  recovery path on macOS in v1** ‚Äî a known open Tauri crash, ¬ß0.4.4 ‚Äî so a mid-stream
  reload surfaces as `AppFault`, not a silent recovery), `open_path` (C9 ‚Äî open-folder/open-file by `OpenTarget` id, ¬ß7.7), `open_project_page`
  (C10 ‚Äî the About "open Releases" link, ¬ß5.9/¬ß7.6), `get_app_info` (C11 ‚Äî About data,
  ¬ß5.9), `get_engine_health` (C12 ‚Äî drives disabled/omitted target tiles, ¬ß5.2/¬ß3.4),
  `cancel_ingest` (C13 ‚Äî the Collecting cancel-collect control, ¬ß5.2/¬ß5.10). The
  frontend treats these as opaque typed RPCs.
- Long-running work (the conversion run) must **not** block on a single Promise
  resolving at the end (Cloudflare-100s-style hangs don't apply locally, but a
  60-minute batch resolving one Promise at the end gives no progress) ‚Äî instead
  the **`start_conversion` command returns quickly** and progress flows over a
  **Channel** (below). The pattern mirrors the platform's "respond immediately,
  stream/poll the rest" posture.

#### IPC call-timing on the Confirm ‚Üí Targets ‚Üí Destination flow `[DECIDED]`
The ¬ß5.2 state machine leaves *which transition* fires C3/C4/C5 implicit; pinned here so
the Phase-3 task list is unambiguous:
- **C3 `get_targets`** is called **on the Confirm ‚Üí Targets transition (3 ‚Üí 4)** ‚Äî the UI
  enters `Targets` only **after** C3 resolves (so the FormatPicker always has its tiles +
  pre-highlighted default before it is shown). C3 is a pure function of detection (no
  engine), so it is cheap.
- **C4 `plan_output`** is called **once a target is selected** ‚Äî i.e. **eagerly on entry
  to `Targets` with the pre-highlighted default already selected**, then **re-called
  (debounced ~150 ms) on any target change or option change** so the `DestinationBar`
  "will save to ‚Ä¶" line, the divert preview, the `rerun` prompt, and the
  `preflight.up_front_fail` verdict never go stale. The SSOT requires a destination preview
  **before** Convert, so C4 must have resolved before the Convert button is enabled.
  **After a C5 destination change, a subsequent target/option change still re-runs C4
  (¬ß0.4.1) ‚Äî but that C4 call carries the C5-resolved destination in its `destination`
  argument (the held session destination, the C5-set `DestinationChoice::ChosenRoot(DestinationId)`,
  not the persisted default), so C4 never resets
  the destination away from the user's C5 choice.** Destination authority stays with C5.
- **C5 `set_destination`** is called **only when the user changes the destination** via
  the C2b `pick_destination` picker (Change-destination); it re-evaluates the
  destination-dependent preflight and **carries `rerun` through unchanged** (¬ß2.5.1
  destination-independence). It is **not** part of the default forward path.

#### Persisted `lastDestinationMode` ‚Üí C4's default destination `[DECIDED ‚Äî re-cut by the 2026-07-06 owner ruling (core-owned paths); the hand-off wire form re-cut at P3.56 (C14 query)]`
¬ß7.4 persists a `lastDestinationMode` prefs key (`"beside-source"` | `"<absolute path
string>"`, default `beside-source`). Who reads it, when, and how it reaches C4 is pinned
here so a returning user's `DestinationBar` initial state is derivable:
- **The stored value is explicitly COSMETIC-LOSSY:** it is a *display-grade hint*, never
  an authority ‚Äî consistent with the ¬ß7.4 cosmetic-persistence posture (persistence is
  best-effort and never load-bearing). At launch/first use the **core re-validates it**;
  a stored path that does not round-trip/resolve (renamed, gone, read-only, a
  non-UTF-8-mangled name ‚Äî ¬ß2.10.1) **falls back to beside-source** (fail-clear, the
  ¬ß2.7 per-location fallback surfacing applies).
- **WHO + WHEN:** the **core reads `lastDestinationMode`** (`crate::prefs` ‚Äî the same
  ¬ß7.5.3 `prefs::load` door that reads `verboseLog`); the **WebView never touches the
  stored path** (the 2026-07-06 owner ruling: no FS path crosses the wire in either
  direction ‚Äî a prefs-stored path is still a path). The frontend holds only the fact
  *that* a persisted choice exists to pre-select, never its value.
- **HOW it reaches C4 ‚Äî the pre-C4 hand-off `[DECIDED ‚Äî P3.56]`:** at the eager `3‚Üí4`
  advance the frontend calls **C14 `get_initial_destination`** (¬ß0.4.1) BEFORE the first
  C4 `plan_output`. Its handler resolves `lastDestinationMode` core-side into a
  **structural `InitialDestination`** (`besideSource` | `{ chosenRoot: DestinationPicked }`
  | `fallback`, ¬ß0.6) ‚Äî **a fact, never a path or a `{ mode: "last" }`-intent-on-C4**:
  `"beside-source"` ‚Üí `besideSource`; a stored absolute path that re-validates ‚Üí a session
  **`DestinationId`** minted in the core destination registry (the same registry C2b/C5
  use), returned as the `DestinationPicked { destination, display }` pair; a path that
  fails re-validation ‚Üí **`fallback`** (a STRUCTURALLY distinct outcome from a plain
  beside-source pref, so the passive fallback note surfaces even when beside-source is
  itself writable ‚Äî only the resolver knows the fallback happened). The frontend then maps
  the outcome onto C4's FIRST `destination` argument ‚Äî an **ordinary `ChosenRoot(id)` or
  `BesideSource`** ‚Äî so ¬ß0.6's 2-variant `DestinationChoice` AND the id-less
  `OutputPlanPreview` stay permanent (**no `Last` variant, no C4 mirror-back field**). The
  stored path never bypasses the no-harm machinery ‚Äî C4's per-volume preflight (¬ß1.10) and
  the ¬ß2.7 writability/divert check run against the *resolved* destination as ever; the
  re-validation-failure fallback note is the ¬ß5.7:825 chrome string the `DestinationBar`
  renders from the `fallback` fact (surfaced independently of the C4 divert note).

### Progress subscription lifecycle (Channel)
Per-item and batch progress stream from Rust via a **`tauri::ipc::Channel`**
(ordered delivery ‚Äî correct for sequential progress) passed *into* the
`start_conversion` command. The payload is a **discriminated union** (the exact
variants/fields owned by ¬ß0.4; illustrative shape):

```ts
// shape OWNED by ¬ß0.4.2 ‚Äî shown to fix the frontend handling, not to re-decide it.
// These are the GENERATED types (¬ß0.4.5 tauri-specta bindings.ts); names/fields
// MUST match ¬ß0.4.2's ConversionEvent enum exactly (no `any`, no drift).
import { invoke, Channel } from '@tauri-apps/api/core';
import { commands, type ConversionEvent } from '@/lib/ipc/bindings'; // ¬ß0.4.5

// ConversionEvent (the ¬ß0.4.2 `#[serde(tag="type", content="data")]` enum) =
//   | { type: 'RunStarted';   data: { runId: RunId; totalItems: number; willReencode: boolean } } // non-optional (Rust `bool`, ¬ß0.4.2)
//   | { type: 'ItemStarted';  data: { runId: RunId; itemId: ItemId; sourceDisplay: string; target: TargetId } }
//   | { type: 'ItemProgress'; data: { runId: RunId; itemId: ItemId; fraction: number | null; stage: JobStage } }
//   | { type: 'ItemFinished'; data: { runId: RunId; itemId: ItemId; outcome: ItemOutcome } } // Succeeded{outputDisplay}|Failed{error}|Skipped{reason}|Cancelled
//   | { type: 'BatchProgress';data: { runId: RunId; done: number; total: number } }
//   | { type: 'RunFinished';  data: RunResult };                       // ¬ß0.6 RunResult

async function startRun(plan: StartArgs): Promise<void> {
  const ch = new Channel<ConversionEvent>();
  ch.onmessage = (m) => store.applyConvertEvent(m);  // updates ProgressList/Summary
  await commands.startConversion(plan, ch);          // typed wrapper over C6 (¬ß0.4.5)
}
```

- **Lifecycle:** the Channel is created when entering `Converting`, its
  `onmessage` reduces into the store's progress map; on `runFinished` (or on a
  confirmed cancel) the machine moves to `Summary` and the Channel is dropped.
- **Throttling for large batches:** the UI coalesces high-frequency
  `itemProgress` ticks into animation-frame updates and relies on store-selector
  granularity so a 1000-row `ProgressList` (virtualised, ¬ß1.10) doesn't re-render
  per tick. SR announcements are throttled separately (¬ß5.6).
- **Run/instance identity** (`runId`, single-vs-multi-instance) is owned by **¬ß7.1**
  ‚Äî the frontend just carries the `runId` the backend assigned.

### Intake scan telemetry subscription (Channel)
Mirrors the progress-subscription lifecycle for the **`onScan` `Channel<ScanProgress>`**
(¬ß0.4.1 C1, ¬ß0.4.2): the frontend **always creates and hands** the `onScan` Channel when it
issues **C1 `drain_intake`** ‚Äî C1 is the **single `onScan` carrier** (C2a walks nothing and
takes no channel, ¬ß0.4.1); it is a **non-optional** argument (¬ß0.4.1;
the "optional" intent is realised by **subscribing only when it wants the count**, not by omitting the
arg) ‚Äî and sets `ch.onmessage = (m) =>
store.setScanCount(m.scanned)` so the ¬ß5.2 *Collecting* "Scanning‚Ä¶ N files so far"
display updates (‚âà2/s, throttled, coalesced into animation-frame updates like the
progress map), and **drops** the Channel when C1 resolves (the `CollectedSet` returns)
‚Äî after which the UI leaves *Collecting*. The Channel is **command-scoped** (not an
`app://` event, so it does not count against the ¬ß0.4.2 three-event invariant). If no
count has arrived yet, *Collecting* shows the indeterminate "looking at your files‚Ä¶"
fallback (¬ß5.2 state 2). **Cancellation:** the ¬ß5.2 cancel-collect control / **Esc**
fires **C13 `cancel_ingest`** with the same `CollectingId` (¬ß5.10) and drops the Channel;
the partial set is discarded Rust-side (¬ß1.1).

### Cancellation ‚Äî optimistic vs confirmed round-trip
SSOT *Visible progress, cancellable* + the cancel **mechanism owned by ¬ß1.7**
(process-group kill, no-partial-leftover) means cancel is **not** instant and
must be honest:

1. User hits Cancel (button or **Esc**, ¬ß5.10) in `Converting`.
2. UI **optimistically** flips to a "Cancelling‚Ä¶" affordance and **disables**
   Cancel (no double-cancel) ‚Äî **a second Esc (or Cancel click) while in the
   "Cancelling‚Ä¶" state is silently ignored** (no second `cancel_run`, no UI change),
   so the user cannot double-cancel; the UI does **not** fabricate a finished state.
3. Frontend calls the **cancel command** (name ¬ß0.4) with the `runId` /
   cancellation token (¬ß0.4 token shape).
4. The backend stops the queue, kills the in-flight engine (¬ß1.7), cleans the
   in-progress temp (¬ß2.6), and emits terminal `ItemFinished{outcome:Cancelled}`
   for the stopped item + `RunFinished` over the **same Channel** (¬ß0.4.2 names).
5. Only on that **confirmed** `RunFinished` does the UI move to `Summary`
   (partial): items already finished are **kept** (SSOT), the in-progress one is
   shown `Cancelled`, no partial leftover. The UI never claims "cancelled" before
   the backend confirms the kill+cleanup landed.

### Backend disconnect / panic / fault handling
Pairs with **¬ß2.13** (app-level fault model) and **¬ß0.9/¬ß1.11** (concurrency/
progress). Frontend behaviour:
- **A single item's engine crash/hang** arrives as a normal
  `ItemFinished{outcome:Failed{error}}` (¬ß0.4.2; ¬ß2.13 catches the worker panic and
  reports it as a clean per-item failure) ‚Äî the UI renders the failed row and
  continues; **not** an app-level event.
- **Channel goes silent / command Promise rejects unexpectedly** (core panic, IPC
  drop): the UI shows a **plain app-level fault** message (no stack trace, ¬ß2.13)
  in place of `Summary` ‚Äî "Something went wrong and the conversion stopped." ‚Äî and
  offers "Start over" ‚Üí `Idle`. It does **not** invent per-item outcomes for items
  it never heard back about.
- **WebView/startup faults** (missing engine binary, damaged bundle, WebView fails
  to load) are **startup** concerns owned by **¬ß7.2/¬ß2.13** ‚Äî outside this state
  machine (the UI may never even mount). This section only handles faults that
  occur **after** the UI is live.

### App-wide event subscription (`app.emit` / `listen`)
The three app-wide events (the **only** `app.emit`/`listen` events ‚Äî ¬ß0.4.2:
`app://fault`, `app://intake`, `app://close-requested`; no others) are subscribed
**on mount** of the root shell (alongside the per-run telemetry Channel and the
per-ingest `onScan` Channel ‚Äî Channels are command-scoped, not `app://` events, so
they do not count against the three-event invariant):
- **`app://fault`** ‚Üí render `AppFaultNotice` (state 12), per the disconnect handling
  above.
- **`app://close-requested`** ‚Üí render `QuitConfirm` (state 11), per ¬ß7.3.
- **`app://intake` (a PAYLOAD-LESS nudge)** ‚Äî emitted by the ¬ß7.8.1 funnel for **every**
  intake source (native drop, C2a-picked set, second-instance launch, Open-with,
  launch-arg; ¬ß7.1/¬ß7.8). It carries **no paths** (the 2026-07-06 owner ruling,
  core-owned paths ‚Äî the real paths live core-side in `PendingIntake`). On receipt,
  call **C1 `drain_intake`** with a freshly-generated `collectingId` + the ¬ß5.8
  `onScan` Channel; the core consumes its `PendingIntake` buffer (consume-once per
  call) and the returned **`CollectedSet` RESPONSE drives ¬ß5.2** (enter `Collecting`,
  then the row-2 exits ‚Äî exactly like a drop always did). The **same drain fires once
  on root-shell mount** (after the `app://intake` listener registration settles,
  ¬ß7.8.1), collecting a first-launch / Open-with set buffered before any listener
  existed; a drain that finds **nothing pending** returns `CollectedSet::Empty` and is
  a clean no-op ‚Äî the machine stays put (the ordinary no-files launch, or a nudge
  whose stash a concurrent drain already consumed; how the response distinguishes
  nothing-pending from a real collected-but-ineligible set is ¬ß0.4.1's contract ‚Äî the
  ¬ß5.2 row-2 `Empty ‚Üí Unsupported` exit applies only to a real collection).
  **Primary gate is the core `[DECIDED]`:** the ¬ß7.8.1 funnel is the authoritative
  refuse-busy point ‚Äî when a run is in flight the **core neither stashes nor emits the
  nudge** (¬ß7.1.1). The UI rule below is **defence-in-depth** for the mid-run case, and
  the *real* surface for the non-run states that cannot take fresh intake. Owned/emitted
  by ¬ß7.8.1; consumed here. **UI consumption rule:** a nudge received in a
  **fresh-intake state** (`Idle`/`Summary`, the pre-run wizard states 3/4/5, or the
  refusal states 9/10 ‚Äî the ¬ß5.4 policy set) triggers the drain; a nudge in any
  **other** state (`Collecting` (2), `RerunPrompt` (6), `Converting` (7)/(7a) ‚Äî
  core-gated, so only a leaked nudge ‚Äî `AppCloseRequested` (11), `AppFault` (12)) does
  **not** drain and **renders the passive `BusyNotice` Banner (¬ß5.3) under the
  `AppHeader`** so the refuse is visible rather than a silent drop (the running batch is
  never interrupted).
  - **BusyNotice trigger ‚Äî simplified by the 2026-07-06 owner ruling `[DECIDED]`.**
    BusyNotice appears on the nudge-while-not-drainable path above. On the
    **core-primary** refuse-busy path (¬ß7.1.1/¬ß7.8.1 `forward_launch_intake`
    busy-drop), the core **does NOT emit the nudge** and therefore **does NOT drive
    BusyNotice** ‚Äî the primary path's visible feedback is the **single-instance

### `RunStarted.willReencode` consumption
`RunStarted` (¬ß0.4.2) carries `willReencode: boolean` (the ¬ß2.9.2 best-effort
worst-case flag). **The generated `bindings.ts` type is non-optional `boolean`** (the
Rust struct field is `bool`, ¬ß0.4.2) ‚Äî there is no `undefined` third state. The core
**always emits a definite value** (`false` for non-video / non-applicable batches ‚Äî
¬ß0.4.2 emission rule); for robustness the frontend still treats any **absent /
`undefined` as `false`**. **The worst-case "may be re-encoded" note is first surfaced
at target choice (state 4):** C3 `Target.lossy` can carry the `video_reencode` lossy
kind (¬ß2.9.2), which the FormatPicker renders as a passive note the moment that target
is selected. `willReencode` is therefore the **during-`Converting` confirmation**, not
the note's first or only source. **The carry-over store field (state 4 ‚Üí 7) `[DECIDED]`:**
the note text is held in a single store field **`pendingVideoReencodeNote: string | null`**
‚Äî **set** (to the ¬ß2.9 `video_reencode` string) when C3 returns `Target.lossy =
video_reencode` for the selected target in state 4, **carried unchanged through states
4 ‚Üí 5 ‚Üí 6 ‚Üí 7** (it survives the FormatPicker ‚Üí DestinationBar ‚Üí optional RerunPrompt
transitions so it does not have to be recomputed), and **read by `ConvertingNote`** in
state 7. On `RunStarted` the store reconciles it with `RunStarted.willReencode`: if `true`
it **keeps** `pendingVideoReencodeNote` (and `ConvertingNote` renders it ‚Äî re-surfacing it
in the `Converting` banner if the carried value was somehow empty); if `false` (or absent)
it **sets `pendingVideoReencodeNote = null`**, which **silently clears** the pre-shown
worst-case note. It is the **same ¬ß2.9 string** throughout (state-4 note and state-7 banner
share one value, never two divergent strings). So the note's lifecycle is: **set at state 4
(C3 `Target.lossy`) into `pendingVideoReencodeNote` ‚Üí carried 4‚Üí7 ‚Üí confirmed (kept) or
cleared (set `null`) by `RunStarted.willReencode`**; the authoritative per-item lossy
outcome still comes from the ¬ß1.12 summary.

**Back-navigation reset (4‚Üí3) `[DECIDED]`.** ¬ß5.2/¬ß5.10 allow a `Targets (4) ‚Üí Confirm
(3)` back-navigation (the Back button / Ctrl/‚åò+Backspace). On that **4‚Üí3** transition the
reducer **resets `pendingVideoReencodeNote = null`** ‚Äî otherwise a re-encode note set for a
*prior* target could persist and mislead if the user re-enters Targets and picks a
**different, non-lossy** target. On the next **3‚Üí4** the field is **re-derived from the
fresh C3 response** (set only if the newly-selected target's `Target.lossy = video_reencode`).
So the field always reflects the *current* selection, never a stale prior one; the same
reset applies to any path that re-runs C3 for a new selection.

> **Safe-default rule.** Where a gate has a destructive-vs-safe choice, **Enter
> activates the safe option** and it is the pre-focused one: the re-run prompt
> defaults to **Skip**, never to overwriting/duplicating. This encodes the SSOT
> no-harm bias directly into the keyboard model.

### Global shortcuts policy `[DECIDED ‚Äî none OS-global]`
ConvertIA registers **no OS-global hotkeys** (no system-wide shortcut grabbing) ‚Äî
all accelerators are **in-app only**, active while the window is focused. This
keeps the portable / no-system-pollution posture (SSOT *Portable*) and avoids the
Tauri global-shortcut permission on the ¬ß0.10 allowlist. Menu accelerators (if a
native menu is added) are app-window scoped.
