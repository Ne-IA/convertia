## 5.1 Frontend stack & structure

### Stack `[DECIDED]`
React 19 + TypeScript (strict, **no `any`** per the platform rule) + Tailwind CSS,
built by Vite, running inside the Tauri WebView. Pinned versions are owned by
**В§0.8** (not re-pinned here). The frontend is **UI only** вҖ” it holds no
conversion logic, no filesystem access, and no engine knowledge; every effectful
operation crosses the IPC boundary into the Rust core (В§0.3 two-tier model). The
WebView runtime varies per OS (WebView2 / WKWebView / WebKitGTK, В§0.3.1) вҖ” the
frontend therefore targets the **intersection** of those engines and avoids
bleeding-edge CSS/JS that drifts across them (rendering-drift testing is В§6.4).

### Component / folder structure `[DECIDED]`
Frontend lives under the physical tree owned by **В§0.7** (`src/`). Logical
grouping inside it:

**Hard rule:** only `src/lib/ipc/**` (the В§0.7 path) imports `@tauri-apps/api`.
Components and state talk to a typed faГ§ade, so the IPC contract (В§0.4) has exactly
one consumer and the "no direct `fetch`/raw-`invoke` in feature code" discipline
(platform auth-pattern analogue) is enforceable by lint.

### State management choice `[DECIDED вҖ” Zustand, recommendation]`
The app is a **single linear wizard with one batch in flight at a time** (the
SSOT batch rule allows only one source format per run; В§1.3). That makes the
state small and mostly a finite-state-machine plus a per-item progress list.

- **Screen-flow state** вҶ’ a hand-rolled **reducer-based finite-state machine**
  (`state/machine.ts`, plain TS, see В§5.2). A typed discriminated-union `State`
  + `dispatch(action)` is preferred over a library (XState would work but is
  weight the SSOT *lightweight* principle doesn't justify for ~9 states).
- **Shared app store** вҶ’ **Zustand** `[recommendation]`. Rationale: tiny, no
  provider boilerplate, ergonomic selectors (avoids needless re-render of a
  1000-row progress list), and trivially testable in Vitest. The store holds the
  machine state, the collected batch, the chosen target+options, the resolved
  destination preview, and the live progress map. Redux Toolkit is heavier than
  warranted; bare Context+useReducer re-renders too broadly for the large
  progress list (В§1.10 virtualisation). **This is the one genuinely
  substitutable decision in 5.1** вҖ” flagged as a recommendation, not owner-level
  blocking; any equivalent minimal store is acceptable so long as it keeps the
  IPC-faГ§ade rule and selector-granularity.
- **No data-fetching/cache library** (React Query etc.) вҖ” there is no server; all
  data crosses IPC and is push-driven via a Channel (В§5.8), not request/cache.

### Shared types with Rust
The RustвҶ”TS type bridge (so `Batch`, `DetectionOutcome`, `ConversionJob`,
`Target`, `RunResult`, error/lossy payloads, the progress event union) are
**typed end-to-end with no `any`** вҖ” mechanism owned by **В§0.4.5** (DECIDED:
**tauri-specta** + specta; CI drift check in В§06). This section **consumes**
those generated types from `src/lib/ipc/bindings.ts` (В§0.7/В§0.4.5) and re-decides
nothing.

## 5.2 Screen states / flow вҖ” **owner of the frontend state machine**

The UI is a finite-state machine. Each state is a discriminated union variant
carrying exactly the data that state needs; transitions are driven by user
actions and by inbound IPC results/events (В§5.8). The backend is the source of
truth for *facts* (what was detected, what targets exist, where output will go,
per-item outcome); the machine only sequences the user through them.

### States (enumerated)

| # | State | Entered when | Primary content | Exits to |
|---|-------|--------------|-----------------|----------|

| 7a | `Converting (CancellingвҖҰ)` (sub-state of 7) `[DECIDED]` | user pressed **Cancel** / **Esc** during `Converting` (C7 `cancel_run` fired, В§5.8 round-trip in flight) | **stays in state 7's screen** вҖ” the **progress list is still live** (already-finished items kept, the in-progress item winding down); the **Cancel button is disabled** with a **"CancellingвҖҰ"** label; a **second Esc is ignored** (no double-cancel, no quit-confirm here); no new control appears | backend confirms cancel вҶ’ `Summary` (partial); the run never returns to interactive `Converting` from here |
| 8 | `Summary` | every job reached a terminal state (В§1.9) | per-item success/fail with reason (strings В§2.8), outputвҶ’source mapping (В§1.12), **Open folder** / **Open file** (OpenActions, В§7.7); **when outputs split (`RunResult.divert_root` is `Some`), TWO open-folder buttons** вҖ” beside-source (`common_root`) + the divert root (Downloads/Documents) вҖ” else a single common-root button (В§5.3/В§7.7.1); a **fully-failed** batch is rendered as a clear failure banner, never a quiet "done" | "Convert more" вҶ’ `Idle`; Open actions stay available |

| 10 | `Unsupported` / `Unreadable` | detection says *real but unsupported type* or *uncertain/conflicting* (В§1.2), or every collected item was unreadable/gone, **or** a `CollectedSet::Empty` (all files filtered out / nothing eligible) | plain message: *"Can't convert this type вҖ” detected: X"* / *"Couldn't tell what this file is"* / *"Nothing here I can convert"* (the Empty case); never an empty target list, never a hang | dismiss вҶ’ `Idle` |
| 11 | `AppCloseRequested` (overlay over `Converting`) | the OS window-close was intercepted mid-run вҶ’ backend emits **`app://close-requested`** (В§0.4.2 / В§7.3.2) | a calm confirm **interstitial over** `Converting`: *"A conversion is in progress. Quit anyway? Files already finished are kept; the one in progress will be discarded."* вҖ” **Quit** / **Stay**; **Enter = Stay** (safe default), **Esc** cancels the close | **Stay** вҶ’ back to `Converting`; **Quit** вҶ’ backend runs cancel+cleanup+exit (В§7.3.3) вҶ’ app exits |
| 12 | `AppFault` | **scoped to the run path `[DECIDED]`:** the `Converting` Channel goes silent or the C6/C7 Promise rejects unexpectedly (core panic, IPC drop during a run вҖ” В§2.13/В§5.8), in place of `Summary`; **plus** the app-level `app://fault` faults (WebView core disconnect, startup engine-missing, damaged bundle). A **pre-run** C3/C4/C5 rejection does **NOT** escalate to AppFault вҖ” it renders as an **inline error in the current state** (e.g. a `Targets`/`Destination` inline `Note`), since no run is in flight and recovery is local (SSOT-faithful "fail clearly" without a full-screen takeover) | plain **no-stack-trace** message: *"Something went wrong and the conversion stopped."* + **Start over**; never fabricates per-item outcomes for items it never heard back about | **Start over** (Ctrl/вҢҳ+N) вҶ’ `Idle` |

> **Mid-run skip vs pre-flight refusal вҖ” keep distinct (SSOT *Fail clearly*).**
> `MixedDropRefusal` (state 9) and `Unsupported` (state 10) are **pre-flight** вҖ”
> nothing is converted. A single bad/unreadable/too-big item discovered **during**
> `Converting` is **not** a state change: it surfaces as that item's terminal
> `Failed`/`Skipped` row in `ProgressList` and carries into `Summary`; the batch
> keeps going (В§1.9/В§2.8). Two visually similar "something's wrong" outcomes, two
> different mechanisms вҖ” the UI must not conflate them.

## 5.4 Drag-and-drop & input вҖ” **the native file-drop boundary (a В§0.4 fact)**

### Native file-drop (the load-bearing constraint)
**HTML5 drag-and-drop inside a Tauri WebView does not expose real filesystem
paths** (this is the В§0.4 boundary fact). Intake therefore uses **Tauri's native
drag-drop event**, not the DOM `drop` event:

```ts
// src/lib/ipc/events.ts вҖ” the ONLY place this is wired (В§0.7 path)
import { getCurrentWindow } from '@tauri-apps/api/window';

const unlisten = await getCurrentWindow().onDragDropEvent((e) => {
  switch (e.payload.type) {
    case 'enter':
    case 'over':  setDragActive(true); break;        // visual affordance only
    case 'leave': setDragActive(false); break;
    case 'drop':  setDragActive(false);
                  // e.payload.paths: absolute path strings (files AND folders).
                  // C1 (В§0.4.1) REQUIRES a frontend-generated collectingId + origin
                  // (README DECIDED: "the frontend generates CollectingId and passes
                  // it as a C1 argument"), so C13 can cancel this in-flight walk.
                  const collectingId = crypto.randomUUID() as CollectingId;
                  ipc.ingestPaths({
                    paths: e.payload.paths,
                    origin: 'drop',          // IntakeOrigin (REQUIRED, В§0.4.1 C1)
                    collectingId,
                    // onScan?: optional ScanProgress Channel (В§5.8 intake telemetry)
                  });                          // вҶ’ В§0.4 C1 ingest_paths
                  break;
  }
});
```

- `event.payload.paths` is an **array of absolute path strings** (not DOM `File`
  objects). Folders arrive as a directory path; **folder recursion runs in Rust**
  (В§1.1) вҖ” the WebView cannot and must not enumerate a directory.
- The window must have `dragDropEnabled: true` (Tauri default) for the native
  event to fire; the DOM-level DnD that *would* hijack it is left disabled.
- `onDragDropEvent` is a method on the **Window** object (`getCurrentWindow()` from
  `@tauri-apps/api/window`) in Tauri v2; verify the exact import/method against the
  pinned Tauri v2 minor at build time (В§6.1) вҖ” the v1 `fileDropEvent` API is gone.
- **Known gotchas to handle (В§6.4 test items):** native drag-drop events can
  duplicate / report differing webview ids on some platforms, and path payloads
  have shifted between Tauri patch versions вҖ” `src/lib/ipc/events.ts` de-duplicates by
  the `paths` set per drop and treats the backend's frozen-set de-dup (В§2.4) as
  the authority; the frontend never assumes the path list is unique or canonical.
- The drop only *hands paths to the backend*; the **frozen source set** (В§2.4),
  detection (В§1.2) and grouping (В§1.3) all happen Rust-side. The UI transitions
  `Idle вҶ’ Collecting` on `drop` and waits for the backend's collected summary.

### File picker (parity path)
Click on the DropZone (or the **Ctrl/вҢҳ + O** accelerator, В§5.10) invokes the **intake
picker command `C2a pick_for_intake`** (`kind: files`). The **choose-folder affordance**
is a **secondary "or choose a folder" link on the DropZone** (and the **Ctrl/вҢҳ + Shift +
O** accelerator, В§5.10) that calls the **same C2a with `kind: folder`** вҖ” it surfaces in
`Idle` next to the files picker, so a folder is as discoverable as a file. The native
dialog is opened **Rust-side via
`DialogExt`** inside the C2a handler `[DECIDED]` вҖ” there is **no JS `open({...})` call
and no `dialog:allow-open` WebView grant** (В§0.10). The picked paths are funnelled
**straight into the C1 `ingest_paths` freeze Rust-side**, and C2a returns the **same
`CollectedSet`** the WebView would get from a drop вҖ” so **no raw FS path ever reaches
the WebView**; the UI transitions `Idle вҶ’ Collecting` and renders the returned summary
exactly as for a drop (one intake funnel, В§1.1). **Cancelling the dialog is a clean
no-op**: C2a returns `CollectedSet::Empty`, the UI stays in `Idle`, no error, no
`Collecting` transition.

The **Change-destination** picker is the *separate* command **`C2b pick_destination`**
(folder-only, also Rust-opened via `DialogExt`): it **returns the chosen folder
`PathBuf` to the WebView**, which carries it into **C5 `set_destination`**. This is the
one picker whose path *does* transit the WebView вҖ” unavoidable, since the destination is
a WebView-held choice вҖ” and is acceptable (a *write* destination, bounded by В§2.1, see
В§0.11 T2a). Capability scope is owned by **В§0.10**; this section only invokes within it.

### Keyboard parity
Every result reachable by drop/pick is reachable by keyboard alone (SSOT DoD
gate). The DropZone is a focusable, `role="button"` element; **Enter/Space**
activates the picker; the full accelerator map is **В§5.10**. There is no
keyboard-only dead end in the flow.

### Launch-time intake
Paths can also arrive via OS launch entry points (Open-with / argv / macOS
open-doc) вҖ” posture owned by **В§7.8**; they feed the **same** `ingest_paths` (C1)
funnel, so the UI handles a launch-with-files identically to a drop (machine
enters `Collecting` at startup instead of `Idle`).

## 5.5 Design system

### Intent
"**Modern > plain**" (SSOT *Design Intent*): uncluttered, contemporary, a little
eye candy вҖ” never busy. Visual polish is **iterative and never release-blocking**
(SSOT В§9 *Not a gate*); the *structure* and tokens here are the contract, the
exact palette is a placeholder the owner finalises (SSOT *Design Intent*: logo,
colours, branding are placeholders for now).

### Tokens `[DECIDED вҖ” token contract; values are placeholders]`
Design tokens are CSS custom properties in `design/tokens.css`, surfaced to
Tailwind via the theme config so components use semantic Tailwind classes, not
raw hex. Token groups:

- **Colour (semantic, not literal):** `--bg`, `--surface`, `--surface-raised`,
  `--border`, `--text`, `--text-muted`, `--accent` (the one brand accent),
  `--accent-contrast`, plus state colours `--success`, `--warn`, `--danger`,
  `--info`. Lossy/divert notes use `--info`/`--text-muted` (calm, **not**
  `--danger` вҖ” predictable loss is not an error). Failures use `--danger`.
- **Spacing scale:** 4-px base (`--space-1`=4 вҖҰ `--space-8`=32) вҖ” Tailwind's
  default scale, kept.
- **Radius:** `--radius-sm/md/lg` (cards, tiles, the DropZone вҖ” generous rounding
  for the modern feel).
- **Typography:** a single clean UI sans (system stack +
  bundled-offline fallback so it renders identically with **zero network**, per
  the offline invariant В§2.11); sizes `--text-xs вҖҰ --text-2xl`; line-heights;
  weight `regular/medium/semibold`. **Readable contrast & text sizes are a DoD
  accessibility gate** (В§5.6), so the *minimum* body size and contrast ratios are
  fixed, not placeholder.
- **Elevation/shadow, motion** tokens (below).

### Light / dark `[DECIDED вҖ” both; follow OS by default; theme persists]`
Support **light and dark**, defaulting to the **OS preference**
(`prefers-color-scheme`), resolved into the colour tokens at the root. **Theme
persistence is `[DECIDED]`:** В§7.4 ships the **3-key prefs blob** (`theme` +
`lastDestinationMode` + `verboseLog`) via `tauri-plugin-store`, so a chosen theme **does** persist
across launches (a `theme` of `system` keeps following the OS). Because the `theme`
key persists, **a minimal in-app Light/Dark/System toggle IS provided** (it writes the
`theme` key via the store, В§7.4.2) вҖ” without one, a persisted non-`system` value could
never be set or reset. (The earlier "no in-app toggle required" line is superseded by
the persistence decision.) The default value is `system` (follow the OS) so the
zero-interaction path is unchanged.

### Motion / eye-candy budget `[DECIDED]`
"A bit of eye candy is welcome" (SSOT) but **restrained and accessible**:
- Subtle transitions on the DropZone (lift/glow on `dragActive`), tile selection,
  drawer open/close, and state transitions (вүӨ200 ms, ease-out).
- The progress bar animates smoothly between real values вҖ” **never** a fake/
  indeterminate crawl on the convert step (SSOT *Visible progress* demands a real
  bar so a long single conversion reads as *working, not hung*). An indeterminate
  spinner is permitted **only** for the brief `Collecting` step.
- **`prefers-reduced-motion` is honoured** (В§5.6): all non-essential animation is
  disabled/reduced; progress still updates its value, just without easing.

### Loading / progress chrome
- Determinate `ProgressBar` (per-item) + aggregate batch bar in `Converting`.
- `Collecting` may show an indeterminate indicator + a cancel-collect control.
- No modal blocking overlay during conversion вҖ” the window stays responsive
  (SSOT *stays responsive regardless of batch or file size*), Cancel always live.

### App chrome layout `[DECIDED]`
A single **persistent slim app-header** (`AppHeader`, В§5.3) frames every В§5.2 state and
is the home for the three otherwise-homeless surfaces: **left** = the `<BrandLogo>`
Ne-IA mark; **right** = the **ThemeToggle** (Light/Dark/System, В§5.5 *Light/dark*) and an
**About/`?`** trigger (opens `AboutDialog`, В§5.9). The **refuse-busy `BusyNotice`** Banner
(В§7.1.1) renders just under this header (top of the workspace) when a mid-run hand-off is
refused. The header is calm and slim so the workspace (DropZone / pickers / progress)
stays the focus; it never adds a second navigation model. The **choose-folder**
affordance lives on the DropZone in `Idle` (see В§5.4), not in the header.

### Ne-IA logo placeholder `[DECIDED вҖ” placeholder]`
The **Ne-IA logo** appears as branding (header/About) per SSOT *Design Intent*.
It ships as a **bundled local asset** (offline; no CDN) behind a single

## 5.6 Accessibility вҖ” **a v1 ship gate, not polish**

SSOT *For anyone вҖ” accessible* and the В§9 DoD gate ("basic accessibility works:
keyboard path + readable contrast/sizes") make this **release-blocking**, on par
with no-harm. Concrete requirements:

- **Full keyboard operability.** Every action вҖ” open picker, confirm batch, pick
  target, open Advanced, change destination, convert, cancel, open folder/file,
  dismiss a refusal, answer the re-run prompt вҖ” is reachable and operable by
  keyboard alone (map В§5.10). No mouse-only affordance exists. The drop area has a
  keyboard-equivalent (the picker).
- **Logical focus order** following the visual wizard order; focus is **moved to
  the new primary element on each state transition** (e.g. **on entering `Targets`
  (3вҶ’4), focus moves to the pre-highlighted (default-selected) tile in the FormatPicker
  radiogroup** вҖ” the `aria-checked="true"` tile that is the active radio, so a
  keyboard user lands on the live default; to the Convert button
  when the destination is shown), and
- **Converting (7) and Collecting (2) focus-on-entry rules `[DECIDED]`** (the two states
  В§5.6 previously left unstated):
  - **Converting (7):** on entry, focus moves to the **Cancel button** (the one actionable
    control during a run) so a keyboard user can cancel without hunting. When the run ends
    and the machine advances to `Summary`, the В§5.6 Summary focus-on-entry rule takes over.
  - **Collecting (2):** focus moves to the visible **cancel-collect control** (the В§5.2
    Collecting cancel that backs C13, В§5.10) if rendered; if no cancel control is shown yet
    (the walk is too brief to surface one), **prior focus is preserved**. **After C13
    returns to `Idle`**, focus lands on the **DropZone** (the primary Idle affordance), not
    left orphaned on the now-unmounted cancel control.
- **Summary (state 8) focus-on-entry rule `[DECIDED]`:** when the run ends and `Summary`
  mounts, focus moves to вҖ” in priority order вҖ” **(1) the first `Failed` row** (announced
  `aria-live="assertive"`) if any item failed; **(2) the `OpenActions` primary button**
  ("Open folder") if **all** items succeeded; **(3) the fully-failed banner's primary
  affordance** (Start over / Convert more) if **every** item failed. For a virtualised
  results list, the target row must be **scrolled into view and rendered before** focus is
  set (never focus a not-yet-mounted virtual row). And
  **trapped inside modals** with **Esc** to close (В§5.10). **Focus-restore-on-close is
  scoped to the modals that HAVE a UI trigger `[DECIDED]`:** **RerunPrompt** (triggered by
  the Convert button in state 4/5) and **AboutDialog** (triggered by the About control in
  `AppHeader`) restore focus to **their trigger** on close. **QuitConfirm has NO UI
  trigger** вҖ” it is raised by `app://close-requested` (an OS close request, В§7.3.2), not a
  UI element вҖ” so on **Stay/Esc** it returns focus to the **active element in the
  underlying `Converting` state** (the Cancel button, or the focused progress-list row),
  **never** "the trigger" (matching the already-correct В§5.10 QuitConfirm row; restoring
  to a non-existent trigger would be a WCAG 2.4.3 Focus Order failure). The focus-trapped
  modals are exactly three (and only these): the **decision dialogs `role="alertdialog"`**
  вҖ” **RerunPrompt** + **QuitConfirm** вҖ” and the **informational dialog `role="dialog"` +
  `aria-modal="true"`** вҖ” **AboutDialog** (informational, not an alert вҖ” В§5.6
  decision-states block). **Accessible name (WCAG 4.1.2 / ARIA 1.2) `[DECIDED]`:** every
  one of the three carries an accessible name via **`aria-labelledby`** pointing at its own
  heading element вҖ” **RerunPrompt** вҶ’ its heading **"Already converted with these
  settings"**; **QuitConfirm** вҶ’ its heading **"Conversion in progress"**; **AboutDialog**
  вҶ’ its **"About ConvertIA"** heading. An `alertdialog` with no accessible name is an
  axe-core a11y-gate (В§6.4.6a) failure, so the two `role="alertdialog"` elements MUST name
  their heading (the headings are the В§5.3 component contract). **`MixedDropRefusal` (state
  9) and `UnsupportedNotice` (state 10) are NOT modals `[DECIDED]`** вҖ” both are **full-screen
  states** (entered by a drop, so no trigger to restore focus to). MixedDropRefusal has its
  own **active re-drop `DropZone`** (the SSOT "re-drop a single format" reading); on entry
  its **heading is announced assertively** (`aria-live="assertive"`) and focus moves to the
  **re-drop `DropZone`**, with a secondary **Dismiss вҶ’ `Idle`**. `UnsupportedNotice` is
  likewise a full-screen pre-flight notice; on dismissing it, focus is **restored to the
  `DropZone`** (drop again), not left orphaned. Neither uses `role="alertdialog"` and
  neither is focus-trapped.
- **Contrast & text size.** Body text and interactive elements meet **WCAG 2.1 AA
  contrast (вүҘ4.5:1 text, вүҘ3:1 large text / UI)** against both themes; the minimum
  body size and the token scale (В§5.5) respect OS text-scaling; nothing critical
  is conveyed by **colour alone** (the lossy note has text + icon, failures have a
  label not just red).
- **Screen-reader announcements** via an ARIA-live region (`a11y/announcer.ts`):
  - `Collecting`/`Confirm`: announce the collected summary ("48 JPG files
    found"). **Confirm-gate assertive string pattern `[DECIDED]`** (a `strings/ui.ts`
    string, announced `aria-live="assertive"` on entering state 3): when **no** items were
    skipped, **"{n} {FORMAT} files"** (e.g. "48 JPG files"); when **M were skipped**,
    **"{n} {FORMAT} files вҖ” {m} file(s) weren't recognized and will be skipped"** (e.g.
    "48 JPG files вҖ” 3 file(s) weren't recognized and will be skipped"). Singular/plural
    handled per `strings/ui.ts`; the same string drives the В§5.2 row-3 visible line and the
    SR announcement (one source, no divergence).
  - `Converting`: announce **batch milestones** (start, each item complete /
    failed *throttled* to avoid a 1000-item flood вҖ” e.g. every N% or on each
    failure), **not** every progress tick. Every `role="progressbar"` (per-item and
    aggregate) carries **`aria-valuemin="0"`, `aria-valuemax="100"`, and
    `aria-valuenow`** (WCAG 4.1.2 вҖ” a ship gate, В§5.6); for the indeterminate-`fraction`
    (LibreOffice) row that has no `aria-valuenow`, set **`aria-busy="true"`** instead so
    SR users hear "busy" rather than a bogus value.
  - `Summary`: announce the outcome ("42 succeeded, 6 failed").
  - **Decision states announce assertively.** Three distinct shapes `[DECIDED]`:
    - **`role="alertdialog"` (focus-trapped DECISION modals)** вҖ” reserved for the genuine
      interrupting, decision-FORCING dialogs: **RerunPrompt** (state 6, Skip/Fresh-copy)
      and the **QuitConfirm** overlay (state 11, Stay/Quit). These force a choice, so the
      "urgent alert requiring action" semantics of `alertdialog` are correct. The **Confirm
      gate** (state 3) is also a required decision point and its summary (incl. any
      skipped-count) is announced assertively on entry, but it is a **normal wizard screen
      state, not a modal** (no focus trap, no `alertdialog`; the assertive announcement is
      via `a11y/announcer.ts`; focus moves to the Confirm button on entry вҖ” В§5.6/В§5.10).
    - **`role="dialog"` + `aria-modal="true"` (focus-trapped INFORMATIONAL dialog)** вҖ”
      **AboutDialog**: it is purely informational (version / licence / verbose-log toggle)
      with a single dismiss, **not** a decision-forcing alert, so `role="alertdialog"` would
      be a **WCAG 4.1.2 (Name/Role/Value) violation** (it would make screen readers announce
      an urgent alert requiring action). It uses `role="dialog"` with `aria-modal="true"`
      and `aria-labelledby` pointing at the "About ConvertIA" heading вҖ” still focus-trapped,
      still Esc-dismissible (В§5.10), only the **role** differs from a decision dialog.
    - **`aria-live="assertive"` heading on a FULL-SCREEN state (NOT a modal at all)** вҖ”
      **MixedDropRefusal** (state 9): it has its own active re-drop `DropZone`, so it is a
      full-screen state, not a trapped modal; its heading is announced assertively but it
      carries no `role="alertdialog"`. **UnsupportedNotice** (state 10, incl. the
      Unsupported / Unreadable / nothing-eligible `Empty` cases) is **likewise a full-screen
      state, NOT a modal `[DECIDED]`**: state 10 is entered by a **drop** (no trigger
      element to restore focus to), the В§5.2 state table treats it like MixedDropRefusal
      (full-screen, dismiss вҶ’ `Idle`), and the В§5.6 dismiss text restores focus to the
      `DropZone` вҖ” all of which are full-screen-state behaviours, not modal ones. So
      UnsupportedNotice carries **no `role="alertdialog"`**; on entry its heading is
      announced assertively (`aria-live="assertive"`) and on dismiss focus moves to the
      `DropZone` (drop again). **AppFault** (state 12) is likewise a full-screen "something
      went wrong" state with an assertive heading and a single Start-over action.
    Lossy/divert notes announce **politely** (`aria-live="polite"`) вҖ” they are calm,
    not alarms.
- **Semantics:** the target tiles are a **`role="radiogroup"`** container with
  **`aria-labelledby`** pointing at its heading; each tile is a **`role="radio"`** with
  `aria-checked` (exactly one default checked, В§1.5); the DropZone is `role="button"`;
  the lossy note is associated with its target via `aria-describedby`. The radiogroup uses
  the standard **roving-tabindex** pattern (only the active/checked radio is `tabindex="0"`,
  the rest `tabindex="-1"`; **arrow keys** move selection within the group, Tab enters/exits
  it). **Disabled (patent-gapped) tiles `[DECIDED вҖ” option (b)]`:** they are **kept in the
  arrow-key navigation order** as `role="radio"` **`aria-disabled="true"`** so a keyboard
  user can arrow onto them and hear the reason text *why* (WCAG 2.1 AA), but they carry
  **`tabindex="-1"`** like every non-active radio вҖ” i.e. they participate in arrow-key
  roving, **not** the Tab sequence (this keeps the roving-tabindex contract intact rather
  than the earlier `tabindex="0"`, which would have broken the single-tab-stop radiogroup
  rule). They are **never** just visual dimming and **never** removed from arrow navigation.
- The В§9 **non-developer walkthrough per platform** is the validation that this is
  *actually* usable, not just spec-compliant.

## 5.8 IPC integration & frontend async model

The **command/event contract is owned entirely by В§0.4** (every command name,
request/response payload, error shape, cancellation token, and every event/
Channel and its payload). This section defines only the **frontend async
behaviour** that *consumes* that contract. Nothing here re-declares a command
name or payload shape вҖ” `src/lib/ipc/commands.ts` and `src/lib/ipc/events.ts` are the
typed wrappers; feature code calls those.

### Command/response model
- All effectful operations are **`invoke()` calls** into the Rust core, awaited as
  Promises, typed via В§0.4.5 generated types (no `any`). The full set of commands the
  frontend calls (names defined in В§0.4, not invented here): `ingest_paths` (C1),
  `pick_for_intake` (C2a вҖ” DropZone "browse" / keyboard picker вҶ’ returns `CollectedSet`)
  and `pick_destination` (C2b вҖ” Change-destination directory picker вҶ’ returns `PathBuf` for C5), В§5.4,
  `get_targets` (C3), `plan_output` (C4), `set_destination` (C5), `start_conversion`
  (C6), `cancel_run` (C7), `get_run_summary` (C8 вҖ” idempotent summary re-fetch by a
  **fresh listener AFTER the run terminated**; **reload-during-run is NOT a supported
  recovery path on macOS in v1** вҖ” a known open Tauri crash, В§0.4.4 вҖ” so a mid-stream
  reload surfaces as `AppFault`, not a silent recovery), `open_path` (C9 вҖ” open-folder/open-file, В§7.7), `open_project_page`
  (C10 вҖ” the About "open Releases" link, В§5.9/В§7.6), `get_app_info` (C11 вҖ” About data,
  В§5.9), `get_engine_health` (C12 вҖ” drives disabled/omitted target tiles, В§5.2/В§3.4),
  `cancel_ingest` (C13 вҖ” the Collecting cancel-collect control, В§5.2/В§5.10). The
  frontend treats these as opaque typed RPCs.
- Long-running work (the conversion run) must **not** block on a single Promise
  resolving at the end (Cloudflare-100s-style hangs don't apply locally, but a
  60-minute batch resolving one Promise at the end gives no progress) вҖ” instead
  the **`start_conversion` command returns quickly** and progress flows over a
  **Channel** (below). The pattern mirrors the platform's "respond immediately,
  stream/poll the rest" posture.

#### IPC call-timing on the Confirm вҶ’ Targets вҶ’ Destination flow `[DECIDED]`
The В§5.2 state machine leaves *which transition* fires C3/C4/C5 implicit; pinned here so
the Phase-3 task list is unambiguous:
- **C3 `get_targets`** is called **on the Confirm вҶ’ Targets transition (3 вҶ’ 4)** вҖ” the UI
  enters `Targets` only **after** C3 resolves (so the FormatPicker always has its tiles +
  pre-highlighted default before it is shown). C3 is a pure function of detection (no
  engine), so it is cheap.
- **C4 `plan_output`** is called **once a target is selected** вҖ” i.e. **eagerly on entry
  to `Targets` with the pre-highlighted default already selected**, then **re-called
  (debounced ~150 ms) on any target change or option change** so the `DestinationBar`
  "will save to вҖҰ" line, the divert preview, the `rerun` prompt, and the
  `preflight.up_front_fail` verdict never go stale. The SSOT requires a destination preview
  **before** Convert, so C4 must have resolved before the Convert button is enabled.
- **C5 `set_destination`** is called **only when the user changes the destination** via
  the C2b `pick_destination` picker (Change-destination); it re-evaluates the
  destination-dependent preflight and **carries `rerun` through unchanged** (В§2.5.1
  destination-independence). It is **not** part of the default forward path.

#### Persisted `lastDestinationMode` вҶ’ C4's default destination `[DECIDED]`
В§7.4 persists a `lastDestinationMode` prefs key (`"beside-source"` | `"<absolute path>"`,
default `beside-source`). Who reads it, when, and how it reaches C4 is pinned here so a
returning user's `DestinationBar` initial state is derivable:
- **WHO + WHEN:** the **frontend reads `lastDestinationMode` from `tauri-plugin-store`**
  (the В§7.4.2 `store:default` mechanism, the same door it uses for `theme`/`verboseLog`)
  **at startup / app-store hydration** вҖ” JS-side via the store plugin, not a new Rust
  command (no IPC surface is added; the store read is the existing capability). It is held
  in the В§5.1 store as the session's default destination.
- **HOW it reaches C4:** the persisted value is the **`destination` argument on C4
  `plan_output`'s FIRST Targets-entry call** (the eager call on the `3вҶ’4` transition above)
  вҖ” `"beside-source"` вҶ’ `DestinationChoice::BesideSource`; `"<absolute path>"` вҶ’
  `DestinationChoice::ChosenRoot(path)`. So a returning user who last chose a custom folder
  sees the `DestinationBar` pre-populated with it (the "will save to вҖҰ" line reflects it)
  without re-picking.
- **A stored absolute path is a re-validated HINT, never a guarantee (В§7.4.1):** C4's
  per-volume preflight (В§1.10) and the В§2.7 writability/divert check re-validate the stored
  path **at use time**; if it has become read-only/gone, the plan **falls back to
  beside-source** (or the В§2.7 divert) and the `DestinationBar` updates to show the
  fallback (a passive note), exactly as a fresh destination change would. The stored path
  never bypasses the no-harm machinery (В§0.11 T2a).

### Progress subscription lifecycle (Channel)
Per-item and batch progress stream from Rust via a **`tauri::ipc::Channel`**
(ordered delivery вҖ” correct for sequential progress) passed *into* the
`start_conversion` command. The payload is a **discriminated union** (the exact
variants/fields owned by В§0.4; illustrative shape):

```ts
// shape OWNED by В§0.4.2 вҖ” shown to fix the frontend handling, not to re-decide it.
// These are the GENERATED types (В§0.4.5 tauri-specta bindings.ts); names/fields
// MUST match В§0.4.2's ConversionEvent enum exactly (no `any`, no drift).
import { invoke, Channel } from '@tauri-apps/api/core';
import { commands, type ConversionEvent } from '@/lib/ipc/bindings'; // В§0.4.5

// ConversionEvent (the В§0.4.2 `#[serde(tag="type", content="data")]` enum) =
//   | { type: 'RunStarted';   data: { runId: RunId; totalItems: number; willReencode: boolean } } // non-optional (Rust `bool`, В§0.4.2)
//   | { type: 'ItemStarted';  data: { runId: RunId; itemId: ItemId; sourcePath: string; target: TargetId } }
//   | { type: 'ItemProgress'; data: { runId: RunId; itemId: ItemId; fraction: number | null; stage: JobStage } }
//   | { type: 'ItemFinished'; data: { runId: RunId; itemId: ItemId; outcome: ItemOutcome } } // Succeeded{outputPath}|Failed{error}|Skipped{reason}|Cancelled
//   | { type: 'BatchProgress';data: { runId: RunId; done: number; total: number } }
//   | { type: 'RunFinished';  data: RunResult };                       // В§0.6 RunResult

async function startRun(plan: StartArgs): Promise<void> {
  const ch = new Channel<ConversionEvent>();
  ch.onmessage = (m) => store.applyConvertEvent(m);  // updates ProgressList/Summary
  await commands.startConversion(plan, ch);          // typed wrapper over C6 (В§0.4.5)
}
```

- **Lifecycle:** the Channel is created when entering `Converting`, its
  `onmessage` reduces into the store's progress map; on `runFinished` (or on a
  confirmed cancel) the machine moves to `Summary` and the Channel is dropped.
- **Throttling for large batches:** the UI coalesces high-frequency
  `itemProgress` ticks into animation-frame updates and relies on store-selector
  granularity so a 1000-row `ProgressList` (virtualised, В§1.10) doesn't re-render
  per tick. SR announcements are throttled separately (В§5.6).
- **Run/instance identity** (`runId`, single-vs-multi-instance) is owned by **В§7.1**
  вҖ” the frontend just carries the `runId` the backend assigned.

### Intake scan telemetry subscription (Channel)
Mirrors the progress-subscription lifecycle for the **`onScan` `Channel<ScanProgress>`**
(В§0.4.1 C1 / C2a, В§0.4.2): the frontend **creates** the `onScan` Channel when it issues
**C1 `ingest_paths`** (or **C2a `pick_for_intake`**), sets `ch.onmessage = (m) =>
store.setScanCount(m.scanned)` so the В§5.2 *Collecting* "ScanningвҖҰ N files so far"
display updates (вүҲ2/s, throttled, coalesced into animation-frame updates like the
progress map), and **drops** the Channel when C1/C2a resolves (the `CollectedSet` returns)
вҖ” after which the UI leaves *Collecting*. The Channel is **command-scoped** (not an
`app://` event, so it does not count against the В§0.4.2 three-event invariant). If no
count has arrived yet, *Collecting* shows the indeterminate "looking at your filesвҖҰ"
fallback (В§5.2 state 2). **Cancellation:** the В§5.2 cancel-collect control / **Esc**
fires **C13 `cancel_ingest`** with the same `CollectingId` (В§5.10) and drops the Channel;
the partial set is discarded Rust-side (В§1.1).

### Cancellation вҖ” optimistic vs confirmed round-trip
SSOT *Visible progress, cancellable* + the cancel **mechanism owned by В§1.7**
(process-group kill, no-partial-leftover) means cancel is **not** instant and
must be honest:

1. User hits Cancel (button or **Esc**, В§5.10) in `Converting`.
2. UI **optimistically** flips to a "CancellingвҖҰ" affordance and **disables**
   Cancel (no double-cancel) вҖ” **a second Esc (or Cancel click) while in the
   "CancellingвҖҰ" state is silently ignored** (no second `cancel_run`, no UI change),
   so the user cannot double-cancel; the UI does **not** fabricate a finished state.
3. Frontend calls the **cancel command** (name В§0.4) with the `runId` /
   cancellation token (В§0.4 token shape).
4. The backend stops the queue, kills the in-flight engine (В§1.7), cleans the
   in-progress temp (В§2.6), and emits terminal `ItemFinished{outcome:Cancelled}`
   for the stopped item + `RunFinished` over the **same Channel** (В§0.4.2 names).
5. Only on that **confirmed** `RunFinished` does the UI move to `Summary`
   (partial): items already finished are **kept** (SSOT), the in-progress one is
   shown `Cancelled`, no partial leftover. The UI never claims "cancelled" before
   the backend confirms the kill+cleanup landed.

### Backend disconnect / panic / fault handling
Pairs with **В§2.13** (app-level fault model) and **В§0.9/В§1.11** (concurrency/
progress). Frontend behaviour:
- **A single item's engine crash/hang** arrives as a normal
  `ItemFinished{outcome:Failed{error}}` (В§0.4.2; В§2.13 catches the worker panic and
  reports it as a clean per-item failure) вҖ” the UI renders the failed row and
  continues; **not** an app-level event.
- **Channel goes silent / command Promise rejects unexpectedly** (core panic, IPC
  drop): the UI shows a **plain app-level fault** message (no stack trace, В§2.13)
  in place of `Summary` вҖ” "Something went wrong and the conversion stopped." вҖ” and
  offers "Start over" вҶ’ `Idle`. It does **not** invent per-item outcomes for items
  it never heard back about.
- **WebView/startup faults** (missing engine binary, damaged bundle, WebView fails
  to load) are **startup** concerns owned by **В§7.2/В§2.13** вҖ” outside this state
  machine (the UI may never even mount). This section only handles faults that
  occur **after** the UI is live.

### App-wide event subscription (`app.emit` / `listen`)
The three app-wide events (the **only** `app.emit`/`listen` events вҖ” В§0.4.2:
`app://fault`, `app://intake`, `app://close-requested`; no others) are subscribed
**on mount** of the root shell (alongside the per-run telemetry Channel and the
per-ingest `onScan` Channel вҖ” Channels are command-scoped, not `app://` events, so
they do not count against the three-event invariant):
- **`app://fault`** вҶ’ render `AppFaultNotice` (state 12), per the disconnect handling
  above.
- **`app://close-requested`** вҶ’ render `QuitConfirm` (state 11), per В§7.3.
- **`app://intake` `{ paths, origin }`** (second-instance launch / Open-with hand-off,
  В§7.1/В§7.8) вҶ’ on receipt, call **`ingest_paths` (C1)** with `{ paths, origin }` and a
  freshly-generated `collectingId` вҶ’ enter `Collecting` (exactly like a native drop).
  **Primary gate is the core `[DECIDED]`:** the В§7.1 single-instance callback is the
  authoritative refuse-busy point вҖ” when a batch is busy (not `Idle`/`Summary`) the
  **core does NOT emit `app://intake`** (the second-launch is refused core-side, В§7.1).
  The UI guard below is **defence-in-depth**, not the primary gate. **UI guard:** an
  `app://intake` received **outside** `Idle`/`Summary` is **ignored** (belt-and-braces
  вҖ” under the core gate it cannot arrive mid-`Converting`; ignoring it elsewhere
  prevents a mid-flow set-swap if the core gate ever regressed). Owned/emitted by
  В§7.8.1; consumed here. **When the UI guard suppresses an `app://intake` received
  outside `Idle`/`Summary`, it renders the passive `BusyNotice` Banner (В§5.3) under the
  `AppHeader`** so the refuse is visible rather than a silent drop (the running batch is
  never interrupted).
  - **BusyNotice trigger вҖ” which path surfaces it `[DECIDED]`.** Resolving the ambiguity:
    **BusyNotice appears ONLY on this UI defence-in-depth guard path** (a leaked
    `app://intake` arriving outside `Idle`/`Summary`). On the **core-primary** refuse-busy
    path (В§7.1.1/В§7.8.1 `forward_launch_intake` busy-drop), the core **does NOT emit
    `app://intake`** and therefore **does NOT drive BusyNotice** вҖ” instead the primary
    path's visible feedback is the **single-instance callback re-focusing the running
    window** (В§7.1.1 `w.set_focus()`), which surfaces the in-flight `Converting` screen so
    the user sees the app is busy. (The core does not need a separate event to show the
    busy note: re-focusing the busy `Converting` UI IS the calm "ConvertIA is busy" signal
    on the primary path; BusyNotice is the belt-and-braces surface for the should-never-
    happen leaked-emit case. This keeps the В§0.4.2 three-event invariant вҖ” no new event is
    added for the busy note.)

### `RunStarted.willReencode` consumption
`RunStarted` (В§0.4.2) carries `willReencode: boolean` (the В§2.9.2 best-effort
worst-case flag). **The generated `bindings.ts` type is non-optional `boolean`** (the
Rust struct field is `bool`, В§0.4.2) вҖ” there is no `undefined` third state. The core
**always emits a definite value** (`false` for non-video / non-applicable batches вҖ”
В§0.4.2 emission rule); for robustness the frontend still treats any **absent /
`undefined` as `false`**. **The worst-case "may be re-encoded" note is first surfaced
at target choice (state 4):** C3 `Target.lossy` can carry the `video_reencode` lossy
kind (В§2.9.2), which the FormatPicker renders as a passive note the moment that target
is selected. `willReencode` is therefore the **during-`Converting` confirmation**, not
the note's first or only source. **The carry-over store field (state 4 вҶ’ 7) `[DECIDED]`:**
the note text is held in a single store field **`pendingVideoReencodeNote: string | null`**
вҖ” **set** (to the В§2.9 `video_reencode` string) when C3 returns `Target.lossy =
video_reencode` for the selected target in state 4, **carried unchanged through states
4 вҶ’ 5 вҶ’ 6 вҶ’ 7** (it survives the FormatPicker вҶ’ DestinationBar вҶ’ optional RerunPrompt
transitions so it does not have to be recomputed), and **read by `ConvertingNote`** in
state 7. On `RunStarted` the store reconciles it with `RunStarted.willReencode`: if `true`
it **keeps** `pendingVideoReencodeNote` (and `ConvertingNote` renders it вҖ” re-surfacing it
in the `Converting` banner if the carried value was somehow empty); if `false` (or absent)
it **sets `pendingVideoReencodeNote = null`**, which **silently clears** the pre-shown
worst-case note. It is the **same В§2.9 string** throughout (state-4 note and state-7 banner
share one value, never two divergent strings). So the note's lifecycle is: **set at state 4
(C3 `Target.lossy`) into `pendingVideoReencodeNote` вҶ’ carried 4вҶ’7 вҶ’ confirmed (kept) or
cleared (set `null`) by `RunStarted.willReencode`**; the authoritative per-item lossy
outcome still comes from the В§1.12 summary.

| Gate | **Enter** | **Esc** |

> no-harm bias directly into the keyboard model.

### Global shortcuts policy `[DECIDED вҖ” none OS-global]`
ConvertIA registers **no OS-global hotkeys** (no system-wide shortcut grabbing) вҖ”
all accelerators are **in-app only**, active while the window is focused. This
keeps the portable / no-system-pollution posture (SSOT *Portable*) and avoids the
Tauri global-shortcut permission on the В§0.10 allowlist. Menu accelerators (if a
native menu is added) are app-window scoped.
